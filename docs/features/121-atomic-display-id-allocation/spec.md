# Spec: Atomic Display-ID Allocation (feature 121)

## Problem

Number allocation is dual and half-racy (PRD "Racy, colliding business key", prd.md:20): the atomic `next_sequence_value` (`database.py:9310`, `BEGIN IMMEDIATE`, kind-generic, keyed `(workspace_uuid, entity_type)`) is used in production only by task promotion and `issue_spawn` (`entity_server.py:582` via `generate_entity_id`, `id_generator.py:62`), while **features get numbers from an unlocked filesystem scan** — `create-feature.md:48` "Find highest number … and add 1" — and the same scan pattern recurs at `decomposing/SKILL.md:158` (features, block-assigned) and `create-project.md:26-31` (projects: "Extract highest NNN, increment by 1"). Two concurrent sessions creating entities of the same kind collide deterministically. Separately: `register_entity` performs **no name validation** (only a `strict`-gated `^\d+-.+` entity_id format check at `database.py:4008/6522`), so blank display fields — which helped corrupt the old DB (P003) — are still insertable today; and the v2 track (118/119, dark) has a `sequences` table (`uuid` PK, NON-unique `(workspace_uuid, kind)` by FR-4 design) but **no allocator over it**, and no rename machinery that records an event (PRD FR-5: "rename/renumber = `update_entity` on display metadata + a recorded event"; roadmap entry 6).

## Scope

Two tracks, mirroring 118/119's live-vs-dark split:

### In scope — LIVE track (v1, ships active)
1. **MCP allocation tool:** new `allocate_entity_id(entity_type, name)` on the entity-registry server → `{seq, entity_id}` in `generate_entity_id`'s existing `{seq:03d}-{slug}` FORMAT, implemented directly against `next_sequence_value` + `_slugify` per the plumbing leaning below (NOT by calling `generate_entity_id` itself — its `project_id`-typed 4th param cannot carry a workspace_uuid). Sequence scope MUST be the server's RESOLVED `workspace_uuid` — NOT the legacy `project_id`/`"__unknown__"` collapse that `register_entity`'s auto_id path inherits (`entity_server.py:575` → `_UNKNOWN_WORKSPACE_UUID`, database.py:5880-5881), which would make two context-less workspaces SHARE a sequence and fail SC1's independence pin. Plumbing leaning (verified): call `next_sequence_value(workspace_uuid=resolved_ws, entity_type=kind)` DIRECTLY (it already exposes the kwarg, database.py:9315) and slugify separately — passing a workspace_uuid THROUGH `generate_entity_id`'s `project_id`-named positional (id_generator.py:62) would hit `_resolve_workspace_uuid_kwargs`'s `project_id_legacy` JOIN and RAISE on no match (database.py:5882-5883). Design finalizes; the contract is workspace-uuid-scoped sequences. Abandoned allocations skip a number — sequence gaps are explicitly acceptable (PRD dismisses "gap anxiety"; values are never reissued). Blank/whitespace name AND name that slugifies to empty (e.g. `"!!!"`) → structured error BEFORE any sequence consumption, via a STANDALONE pre-check (`_slugify(name)` truthiness) ahead of ANY `generate_entity_id` call — `generate_entity_id` consumes the sequence (id_generator.py:62) BEFORE it slugifies (:63) and falls back to `"unnamed"` (:65-66), so routing validation through it would mint `NNN-unnamed` WITH a consumed seq and silently fail the pin. (Both rejection classes are independent SC pins — FR-5 validates "name/slug", two halves; name-non-emptiness does NOT subsume slug-non-emptiness.)
2. **Kill the filesystem-scan allocators (all three):**
   - `create-feature.md` "Determine feature ID" step calls the MCP tool (scan text deleted — FR-5 acceptance is literal: the scan "no longer exists").
   - `decomposing/SKILL.md` Stage 5: per-feature MCP allocation replaces the scan+block-assignment, AND the mapping table/`depends_on` remap consume the tool's RETURNED `entity_id` end-to-end — Stage 5's local slug-derivation prose (SKILL.md:161-166) is deleted so there is exactly ONE slugify source (`id_generator._slugify` truncates on a hyphen boundary, id_generator.py:33-37; the local prose rules don't — divergent slugs would silently break the topological remap).
   - (Project-kind allocation — `create-project.md:26-31`'s `P{NNN}` scan — is explicitly OUT, see Out-of-scope: the shared v1 bootstrap seeds sequences by `re.match(r"^(\d+)", entity_id)` (database.py:9368), and projects register as `P`-leading ids (`P{NNN}-{slug}`, feature_lifecycle.py:275-278, documented as sitting outside the numeric-prefix contract) — the regex never matches, so a v1 rewire would seed `max_seq=0` and deterministically re-mint `P001`, re-creating the PRD's own headline collision. Fixing that means kind-aware format parsing inside a v1 allocator whose table 132 rebuilds anyway.)
3. **Registration validates non-empty name:** `register_entity` (and its `upsert_entity` sibling) reject empty/whitespace-only `name` with `ValueError`; the MCP layer translates to the structured-error envelope consistent with feature-109 §3.5 conventions. The pre-existing truthy-only guard at `entity_server.py:580-581` (`if not name` → plain-string error, auto_id path only, passes `"   "`) is subsumed by the new single-sourced validation — blank-name handling must not fork across the allocate/register surfaces. The validation lives at the DB layer so ALL name-writing paths inherit it — register, upsert, AND `update_entity`'s name-supplied branch (design D2; `name=None` = not-updating, absent ≠ blank).

### In scope — DARK track (v2, guard-allowlisted, wired live at 132)
4. **v2 allocator:** `next_display_seq(conn, *, workspace_uuid, kind) -> int` over v2 `sequences` — atomic single-writer (NFR-1: "Display-id allocation stays atomic. No new concurrency machinery."). Unlike 119's `append_event` (INSERT-only, safe under any caller transaction), this is READ-MODIFY-WRITE: compose mode carries an explicit precondition — the composing caller MUST already hold the write lock (its open transaction is `BEGIN IMMEDIATE`) and owns retry; standalone mode takes `BEGIN IMMEDIATE` + shared retry itself. The `append_event` analogy does not transfer unmodified (a DEFERRED-composing caller could hit `SQLITE_BUSY_SNAPSHOT` on lock upgrade with no retry) — the precondition is part of the contract, docstring-pinned and contention-tested. Must behave deterministically under the schema's deliberate NON-unique `(workspace_uuid, kind)` (FR-4 forbids UNIQUE on business keys): serialization comes from the transaction, not a constraint; exact duplicate-row strategy is design D-2.
5. **v2 rename helper:** updates v2 display fields (`type_id`/`name` — the v2 schema has no separate display_id column; display identity IS `type_id`+`name`, schema_v2.py:49-50) **and** appends a rename event via 119's `append_event` in the SAME transaction (compose pattern — the rename helper holds the write lock, satisfying item 4's precondition class). Identity (`uuid`), FKs, and prior events untouched — PRD acceptance prd.md:70.
6. **Ships-dark guard extension:** the new v2 module joins `_V2_DARK_MODULES` with the established teeth-test pattern (seeded-offender; THREE import spellings incl. the relative `from .display import` — 119's needle set is three-way).

### Out of scope (owning feature)
- 132 cutover: wiring live creation/rename paths to the v2 allocator/rename helper; v1 `sequences` data migration/backfill.
- Display FORMAT rendering (`F-1042`-style kind-prefix presentation) anywhere in live UI/commands — the PRD names the format as an example; whether v2 `type_id` composition changes from `{seq}-{slug}` is a design decision (D-1), but no live surface renders a new format before 132.
- **Project-kind allocation (`create-project.md:26-31`):** deferred to the 132 cutover with a filed checklist item (backlog-manual #054) — rationale above (v1 bootstrap regex-blind to `P`-leading ids; the live `P001`×3 evidence is CROSS-workspace, which per-workspace sequences do not and should not prevent — cross-workspace display collisions are allowed by design, prd.md:104; same-workspace concurrent `create-project` is a negligible-frequency interactive flow). 132's backfill seeds v2 sequences for EVERY kind from the census, making project allocation atomic at cutover with zero v1 surgery.
- Backlog numbering (`docs/backlog-manual.md` is interim-manual while #060 is open; no counter exists to make atomic).
- Brainstorm ids (timestamp-stems, no counter, no race).
- `create-specialist-team.md:109` and `specify.md:19` ("highest" hits that SELECT an existing feature — not allocators).
- Renumber tooling/UI (FR-5 mentions renumber as the same write-path class as rename; the dark helper covers display-field updates generally — no dedicated renumber command ships).

## Error & Boundary Cases

- Blank name OR empty-slug name to the MCP allocate tool → structured error, sequence NOT consumed (validation precedes allocation; two independent branches, each tested — see SC1).
- Concurrent allocation same workspace+kind → distinct values (SC1/SC4); different workspaces → independent sequences (SC1's named cross-workspace test).
- v1 `sequences` row missing for a kind → existing bootstrap-from-entities scan inside the SAME `BEGIN IMMEDIATE` (unchanged, database.py:9356-9377). Pre-backfill safety for the FEATURE kind: the bootstrap sees DB rows, not the filesystem — feature entity_ids are numeric-prefixed so `^(\d+)` parses them (the PROJECT kind fails on id FORMAT, not DB coverage — why it's out of scope). This bootstrap is NOT a rare edge: the live `sequences` table has ZERO feature rows (prd.md:20), so it fires on the FIRST post-rewire allocation in every workspace, and its correctness rests on `entities` holding every on-disk feature for that workspace. Design must pick and cite a stance: a cheap pre-allocation sanity assertion (per-workspace DB feature count ≥ filesystem feature-dir count, fail loud on drift) OR an explicit delegation to 132's backfill-completeness gate tolerating an at-most-once re-mint on a drifted workspace.
- Duplicate `(ws,kind)` rows in v2 sequences (constructible by FR-4 design) → allocator strategy must stay monotonic and non-reissuing (design D-2 picks the strategy; SC4 pins it).
- Compose-mode caller NOT holding a write lock (DEFERRED transaction) → outside the contract (documented precondition, item 4); the helper's docstring names it and design decides whether to assert `conn.in_transaction` + fail loud or trust the docstring.
- Rename of a uuid with NO prior events → valid; SC5 pins stream 0→1 (no ordering precondition).
- MCP degraded/DB unavailable → allocate tool follows the server's existing degradation envelope; `create-feature.md`'s failure step must STOP (a feature cannot be created without an id — warn-and-continue is not acceptable for allocation).

## Open Decisions (design resolves)

- **D-1 (v2 display composition):** does v2 `type_id` stay `{seq:03d}-{slug}` (continuity with every live consumer) or move to kind-prefixed `F-1042` (PRD's example format)? Leaning: keep `{seq}-{slug}` as stored `type_id`; treat `F-1042` as a FUTURE rendering concern (PRD lists namespace-prefixed display ids under Future, prd.md:177) — but design must reconcile this against FR-5's literal example with a named decision.
- **D-2 (v2 duplicate-row strategy):** `MAX(current_value)` across matching rows + update-all vs single-row-by-convention (INSERT only when absent inside the locked transaction, tolerate-and-repair if duplicates appear). Leaning: single-row-by-convention with `MAX()` read — smallest surface, no repair machinery.
- **D-3 (blank/empty-slug rejection layering):** the DB-layer `ValueError` + MCP pre-check are committed (SC1/SC3); design decides whether `generate_entity_id`'s internal `"unnamed"` fallback (id_generator.py:65-66) is also removed — its two live callers (issue_spawn, task_promotion) pass real names; check whether any test pins `"unnamed"` before deleting. The `entity_server.py:580-581` truthy guard is subsumed either way (SC3).
- **D-4 (rename event vocabulary):** `event_type="renamed"`, axis=`lifecycle` (rename is neither pipeline nor execution progress) with `from_value`/`to_value` carrying old/new `type_id` (or name) — design confirms against 119's FR-11 key registry and the three-axis CHECK.
- **D-5 (kind gating):** with project-kind out of scope, does the allocate tool REJECT `entity_type="project"` loudly (preventing a caller from minting a colliding seq before 132 wires the P-aware path) or accept any kind string like `next_sequence_value` does? Leaning: reject "project" with a structured error naming the 132 deferral — one guard line, prevents the exact foot-gun this spec just documented.

## Verification Approach

Tests beside the existing suites: MCP tool tests in the entity-server test module (same-workspace concurrency AND cross-workspace independence via the established multi-connection/multiprocessing patterns; name-half and slug-half rejection each with a no-sequence-consumed assertion); DB-layer validation tests in `test_database.py`; dark-track tests in a new `test_<module>.py` beside `test_events.py`, reusing `test_events.py`'s compose-vs-standalone idiom (TestAppendEventTransactionComposition, :263) and `test_schema_v2.py`'s `_reset_ddl_registry` fixture (:90) + 30-trial multiprocess harness (:708-709, the bootstrap_v2 lock-contention pattern) for SC4's concurrency + compose-contention pins; guard teeth in `test_schema_v2.py`'s established pattern. Command/skill rewires verified by grep-based SC2 assertions plus manual read (command files are instructions, not executable — the enforceable pin is the scan text's absence and the MCP-tool instruction's presence).
