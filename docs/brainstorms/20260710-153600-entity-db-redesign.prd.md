## Problem

The current data model generates its own fragility:

- **Split identity.** `uuid` is PK and half the schema joins on it (relations, tags, deps, OKR, parents); `workflow_phases` and `phase_events` join on the business key `type_id` instead. Two halves, two keys.
- **Fragile business key.** `type_id = "{kind}:{number}-{slug}"` with `UNIQUE(workspace_uuid, type_id)`. Number allocation is split: an atomic DB sequence used ONLY by task promotion (live `sequences` table has zero feature rows), while features get numbers from `create-feature.md:48` "find highest number in features/ and add 1" â€” an unlocked filesystem scan; concurrent sessions collide deterministically. 7 cross-workspace type_id collisions already live in the DB (`project:P001` in 3 workspaces).
- **Phase/status conflation.** `entities.status` is free TEXT with NO CHECK (10 live values incl. 13 empty strings; `active` and `in_progress` coexist). `workflow_phases.workflow_phase` CHECK is the union of FOUR state machines' vocabularies in one enum (feature pipeline / brainstorm lifecycle / backlog lifecycle / 5D). `kanban_column` is derived by `derive_kanban(status, phase)` (`kanban.py:25`) â€” conflation codified. State lives in ~6 places (status, workflow_phase, kanban_column, last_completed_phase, metadata.phase_timing, .meta.json).
- **Dual sources of truth.** 96% of the 601 session-start doctor warnings (579) are DB â†” `.meta.json`/filesystem projection drift. Worse: the largest bucket (320 `entity_orphans`) is substantially FALSE POSITIVES â€” doctor checks still query `entity_type`/`project_id`, columns dropped by migrations 11/12, with the `sqlite3.Error` silently swallowed (`checks.py:1402`); `feature:114` is in the DB as `completed` yet reported "no entity in DB".
- **Defense budget.** 12 of 21 doctor checks defend identity / workspace-scoping / phase-status-sync â€” surfaces this redesign removes. 22 copy-rename table rebuilds across the migration chain; schema version tracked in 3 inconsistent places (`user_version`=0, `_metadata.schema_version`=17, `EXPORT_SCHEMA_VERSION`=1).

## Requirements (user-stated, binding)

- **R2.1 â€” DB is the single source of truth.** Any local state file (e.g. `.meta.json`) is strictly a DB projection, regenerated by the DB layer, never hand- or LLM-edited. ALL workflow management goes through the DB (MCP tools) â€” never through file edits.
- **R2.2 â€” Event table for all status changes.** Every state mutation is an appended event; full audit trail reconstructible from the event log alone.
- **R2.3 â€” Multi-workspace management** is first-class (7 live workspaces today; one global DB).

Plus directions accepted from the review:

- **R2.4 â€” Single identity.** UUIDv7 (time-sortable) as the ONLY key everywhere; `workflow_phases`-equivalent state and events key on it too. `kind:number-slug` becomes display metadata: mutable, non-unique, human-facing only. No uniqueness constraint on it; collisions are cosmetic.
- **R2.5 â€” Two orthogonal state axes.** `pipeline_phase` (authoring: brainstormâ†’specifyâ†’designâ†’create-planâ†’implementâ†’finish; NULL for non-authored kinds; per-kind machine constrains transitions) and `execution_status` (Kanban: backlog / ready / wip / blocked / done â€” one small universal enum with a real CHECK). Kanban board renders `execution_status` directly; card badge renders `pipeline_phase`. `derive_kanban` is deleted.

## Design Direction

### Core model
- `entities(uuid PK [UUIDv7], workspace_uuid FK, kind, display_id, name, created_at, metadata)` â€” no status column at all (see events).
- `events(id, entity_uuid FK, event_type, axis [pipeline|execution|lifecycle], from_value, to_value, actor, timestamp, payload JSON)` â€” append-only, the ONLY write path for state (R2.2). Generalizes today's `phase_events`.
- **Current state = projection.** Either a `state` table maintained transactionally with each event append, or a view over latest-event-per-axis. Reads are served from the projection; the projection is rebuildable from events (`reconcile` = replay).
- `.meta.json` (if it survives at all) is a read-only projection file written solely by the DB layer for fast hook reads (R2.1); the data-file-guard hook keeps blocking other writers. Consider dropping it entirely if session-start can query the DB fast enough.
- Relations (parent, dependencies, fixes, OKR alignment, tags) all keyed on uuid â€” unchanged in spirit, unified in key.

### Display identity (replaces type_id)
- `display_id` like `F-1042` / `B-311` per workspace, allocated from the existing atomic `next_sequence_value` (extended to all kinds â€” kill the filesystem-scan allocation in `create-feature.md`). Purely presentational: renaming/renumbering never touches identity, FKs, or events.

### Multi-workspace (R2.3)
- `workspaces(uuid PK, root_path, name, created_at, last_seen_at)` stays; workspace resolution precedence unchanged (env > flag > workspace.json > recovery). Split-brain fails loud (keep feature-108 behavior).
- Cross-workspace links become ordinary uuid references â€” the allowlist table and its doctor checks dissolve. Workspace filtering happens at query level (`WHERE workspace_uuid = ?` or explicit `--all-workspaces`).
- Multi-workspace views (list/board/lineage) accept a workspace scope parameter; UI gets a workspace switcher.

### Migration strategy
- Private tooling, no back-compat: **rebuild, don't migrate.** New DB file (schema v2 counter reset, ONE version location), backfill by replaying: existing entities â†’ `entity_created` events with fresh UUIDv7s; existing phase_events map onto the new axes; status/kanban derived once at import via the current `derive_kanban` logic, then that logic is deleted. Keep the old DB read-only for 30 days as escape hatch.
- Doctor shrinks accordingly: drop the 12 identity/workspace/sync checks whose surfaces disappear; fix-or-delete the rotted checks querying dropped columns (`checks.py:709,988,1083,1391-1398`) â€” this is worth doing IMMEDIATELY regardless of the redesign, since 320 of the 601 current warnings are spurious and train the user to ignore the doctor.

## Non-goals
- No changes to the phase vocabulary itself (workflow track owns that; the canonical 6 phases stay).
- No multi-writer/concurrent-DB support beyond SQLite WAL + busy_timeout (current concurrency posture is adequate).
- No cloud sync.

## Success criteria
- One key type (uuid) across every table; zero joins on business keys.
- `SELECT` of any entity's full history = one query over `events`; state reproducible by replay (property test: replay(events) == projection).
- Doctor session-start warnings for a healthy workspace: 0 (from 601). Doctor check count materially reduced.
- Concurrent `create-feature` in two sessions/workspaces: no collisions, no user-visible conflict handling.
- Kanban board and phase queries read different columns with no translation function.

## Sequencing

The phase vocabulary is stable (canonical 6 phases preserved per workflow-track R1.2), so nothing blocks on the workflow redesign.
