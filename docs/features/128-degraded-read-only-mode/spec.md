# Spec: Degraded Read-Only Mode (feature 128)

## Problem

PRD FR-10 (prd.md:88): when the DB is unavailable, state MUTATIONS must fail loud and READS serve the last projection — but today `complete_phase` silently degrades to WRITING `.meta.json` (`_write_meta_json_fallback`, engine.py:470, called from the pre-detected-degraded branch at :166 and the `sqlite3.Error` branch at :187). A degradation path that writes reopens the dual-truth hole P004 exists to close: state diverges between a file the DB never saw and a DB the file contradicts. `transition_phase` has TWO degraded branches (engine.py:96 pre-detected; :107-115 mid-write sqlite3.Error catch) — both skip the DB write and return a success-shaped `TransitionResponse(degraded=True)`. At the MCP layer the picture is asymmetric (corrected at skeptic iteration 2): the transition transaction guard (workflow_state_server.py:925-928) raises sqlite3.OperationalError on degraded=True, which the `_with_error_handling` decorator (:733-751) ALREADY converts to a structured `db_unavailable` envelope — so MCP transitions fail loud today; the complete_phase twin guard (:1191-1194) is DEAD CODE — `getattr(completion, 'degraded', False)` on a result type with no `degraded` field is permanently False (verified) — so MCP completes silently WRITE THE FILE. The actual FR-10 violations are: (a) complete_phase's fallback file write; (b) the ENGINE layer's success-shaped degraded returns (any direct engine caller sees silent no-ops); (c) entity_engine's 5D silent degrades (deferred to 123 — see Scope).

## Scope

**In:** the FROZEN engine's mutation-path degradation contract (`complete_phase` both defenses incl. the finish-status sync; `transition_phase` BOTH degraded branches, engine.py:96 and :107-115); deletion of `_write_meta_json_fallback` and its tests; the doctor write-audit allowlist entries naming it (test_audit_writes.py:62, :386); the MCP layer needs NO new mapping — the typed error SUBCLASSES `sqlite3.OperationalError` (pinned here, resolving OQ-2's other half), so the existing `_with_error_handling` decorator (:733-751) already envelopes it as `db_unavailable`; the dead complete_phase guard (:1191 — a two-branch ternary, BOTH branches permanently False: `getattr(completion, 'degraded', False)` on a result type with no such field, and `getattr(state, '_degraded', False)` on FeatureWorkflowState which has no `_degraded` either — both verified) is deleted, while the transition guard (:925) is RETAINED with a dated comment — it stays LIVE for entity_engine's 5D `_fived_transition` degraded=True returns until 123 rebuilds that layer; entity_engine's frozen-engine DELEGATION path (inherits the raise for free) and its :178 cascade-skip-on-degraded (goes dead for mutations post-128 — design confirms and deletes if so).

**Out:** entity_engine's 5D DIRECT mutation paths — these are KNOWN FR-10 violations DEFERRED to 123, not clean: `_fived_complete` CATCHES sqlite3.Error and returns None (entity_engine.py:459-464); `_fived_transition` catches and returns a DB_ERROR-blocked result with degraded=True (:559-574), whose loudness today comes ONLY from the retained :925 MCP guard. Feature 123's unified-engine rebuild replaces this layer wholesale; 128 fixing it would be rework of code 123 deletes. The deferral and the :925 retention are the paired mitigation; data-file-guard rewire and the projection-writer activation (127); read-path redesign (FR-10 KEEPS degraded reads); UI; doctor check retirement (133). Projection chain stated once: 126 derives (dark) → 127 wires the DB-layer projection writer + guard rewire → 132 backfills/cutover.

## Functional Requirements

- **FR128-1 (writer deletion):** `_write_meta_json_fallback` and BOTH call sites are deleted. Repo-wide grep for the symbol returns 0 hits in live code and tests (docs/git history exempt). No replacement fallback writer of any shape is introduced — FR-10's "and equivalents" clause: the deletion PR adds no new code path that writes state files when the DB is down.
- **FR128-2 (mutations fail loud):** with the DB unavailable, the frozen engine's mutation paths — enumerated exhaustively: `complete_phase` pre-detected-degraded branch (:160), `complete_phase` mid-write `sqlite3.Error` catch (:181, which also covers the finish-status sync at :180), `transition_phase` pre-detected branch (:96), `transition_phase` mid-write catch (:107) — raise ONE typed exception class (name pinned at design, e.g. `WorkflowDBUnavailableError`) whose message names the operation, the feature ref, and a recovery hint (doctor / cleanup-locks — exact text design-pinned). RETRY-INTERACTION CONSTRAINT (specify gate): because the class subclasses `sqlite3.OperationalError` and the MCP handlers also wear `_with_retry`, whose `is_transient()` retries any OperationalError with "locked" in the message (sqlite_retry.py:24 — verified), the typed error's message MUST NOT contain the substring "locked" (case-insensitive; "cleanup-locks" is safe — "locks" ≠ "locked" — but e.g. "database is locked" would silently retry a permanent failure ~0.6s — backoff 0.1+0.5 under the call sites' default max_attempts=3, which never reaches the 2.0 tier; corrected at gate round 2); a test pins `not is_transient(err)` on the constructed error. Atomicity of the two-call finish sync (update_workflow_phase :172 + update_entity :180): at the MCP surface the complete_phase handler ALREADY wraps both in one transaction (workflow_state_server.py:1126 — :896 is transition's, corrected at skeptic iteration 2); the engine layer alone has no boundary, and its only non-MCP callers are tests (design re-verifies that claim). SC2's fault-injection exercises the MCP path at :1126 and asserts rollback.
- **FR128-3 (no silent no-op, structured at the surface for free):** both `transition_phase` degraded branches raise the same typed error instead of returning success-shaped degraded responses. The typed error subclasses `sqlite3.OperationalError`, so the EXISTING `_with_error_handling` decorator (:733-751) envelopes it as `db_unavailable` + its standing recovery hint with ZERO new server code — the operation, feature ref, and doctor/cleanup-locks hint travel in the exception MESSAGE (rendered into the envelope's message field). Guard disposition: the dead complete_phase guard (:1191) is DELETED; the transition guard (:925) is RETAINED with a dated 123-handoff comment (still live for 5D degraded returns — see Scope). A server-level test pins the envelope for both mutations (SC5).
- **FR128-4 (reads survive):** the `.meta.json` fallback READ paths — `_read_state_from_meta_json` (def engine.py:446, called from get_state) and the list-operation filesystem fallbacks (`_scan_features_filesystem` def :556, `_scan_features_by_status` def :570), all source="meta_json_fallback" — are UNCHANGED — degraded reads keep serving the last projection, and read-only MCP tools (`get_phase`, `get_progress_view`, …) keep their `degraded` flag behavior. Zero behavior change on the healthy-DB path for everything.
- **FR128-5 (audit teeth retained):** the doctor write-audit allowlist drops the deleted writer's entries — after which the audit red-flags ANY `.meta.json`-writing code reappearing in the engine (the teeth outlive the writer they used to sanction).
- **FR128-6 (test sweep, non-vacuous):** file-level churn inventory (skeptic-iteration-2 measured): test_engine.py (12 `_write_meta_json_fallback` refs incl. the Task-2.2 class + ~85 lines of degraded-write assertions), test_entity_engine.py (1 cascade-skip test, `test_degraded_mode_skips_cascade` :319-355 — 0 `_write_meta_json_fallback` symbol refs; corrected at skeptic iteration 3), test_audit_writes.py (2 allowlist entries), engine.py (3), workflow_state_server.py (guards), models.py (degraded-field docstring naming the sole remaining producer — 5D `_fived_transition` until 123); plan.md refines to test-name granularity. New tests pin the NEW contract with facts FALSE on the old path: (a) mutation under DB-down raises the typed error AND `.meta.json` is byte-identical before/after the attempt (the old path modified it); (b) transition under DB-down raises (the old path returned success-shaped); (c) degraded READ still serves the projection. Every test asserting the old fallback-write behavior is deleted or inverted in the same change — suite green with zero survivors of the old contract.

## Error & Boundary Cases

- **DB recovers between degraded get_state and the mutation:** current code fallback-WRITES even though the DB is healthy again. Design decides: re-probe-and-proceed vs unconditional raise (OQ-1) — either way, no `.meta.json` write.
- **`.meta.json` missing/corrupt during degraded READ:** unchanged behavior (existing read-fallback error handling) — pinned by an existing-or-new test, not redesigned here.
- **DB down at server START:** workflow_state_server.py:192 already logs degraded startup; read tools work, mutation tools return the structured error (SC5).
- **Mid-write failure AFTER partial DB mutation:** `update_workflow_phase` + the finish-status sync must not leave half-state — design pins the transaction boundary (single connection/transaction or ordered-with-idempotent-retry).
