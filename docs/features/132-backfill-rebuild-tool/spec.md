## Scope model (load-bearing)

The rebuild tool creates the new file in three ordered steps — the union DDL is derived BY CONSTRUCTION, never hand-copied:
1. **Replay the v1 MIGRATIONS chain 1→19 on the fresh file.** This yields the exact live v1-operational DDL — entities' polymorphic M14-rebuild shape with the type/kind CHECK (database.py:5110-5133), the 4-vocabulary workflow_phases CHECK incl. 5D/brainstorm/backlog phases (database.py:753-777), phase_events with M19's `cascade_ready` + the `source IN ('live','backfill')` CHECK (database.py:5885-5888, :5224-5239 pre-M19 base), entities_fts + sync triggers, 124's cascade trigger, entity_relations (`('fixes','blocks')` CHECK) — including every index/trigger the chain registers. v1 access-layer code keeps reading/writing these shapes against the new file.
2. **Seed the v2 event core SELECTIVELY** — events + entity_axis_state/entity_state views (120) + the 122 vocabulary triggers (axes.py events_vocab_pipeline/execution, ACTIVE from birth — reconciling 120/126's dark-suite arbitrary-value writes in the same change, roadmap:49) + the _metadata rows. NOT a wholesale bootstrap_v2 (schema_v2.py:130) call: its `IF NOT EXISTS` DDL_REGISTRY would silently skip chain-shaped tables (sequences/workspaces/entity_relations keep their step-1 chain shapes at 132; FR132-5d seeds sequence ROWS into the chain-shaped table); its stripped entities shape (schema_v2.py:43-58, deliberately status-less) is NOT the emitted entities table at 132 — P004 FR-4's column-strip is explicitly deferred to later v1-retirement work. Design pins the connection object per step.
3. **Stamp `schema_generation='v2'`** — after which the FR132-1 guard makes every subsequent open's `_migrate()` a no-op on this file.

Post-cutover, state mutations remain on the v1 write path (which the triggers police) AND land as events via FR132-4b's owned dual-write; per-consumer v2-read rewires beyond this spec's inventory are NOT in scope (133 owns doctor; later work owns v1-path retirement). The old file is retained read-only for 30 days as the escape hatch.

## Functional Requirements

- **FR132-1 (rebuild tool + version bookkeeping):** A committed, idempotently re-runnable rebuild tool stands up the NEW DB file (path distinct from `~/.claude/pd/entities/entities.db`) via the three-step Scope-model build. Version bookkeeping: `_metadata` carries `schema_generation='v2'` + `schema_version=V2_SCHEMA_VERSION` (schema_v2.py:16) as the ONE version location. Because `EntityDatabase.__init__` unconditionally runs `_migrate()` (database.py:6210-6215) which replays `range(current+1, max(MIGRATIONS)+1)` (database.py:10101-10112), `_migrate()` gains an IN-SCOPE generation guard: a `schema_generation='v2'` file returns immediately — the tool's own step-1 chain replay is the sanctioned initial use (it runs BEFORE the step-3 stamp), and the guard protects every open thereafter. (Leaving the file at v1 `schema_version=19` without the marker was rejected: future v1 migrations 20+ would silently auto-run against the v2 file.) New-file fingerprint = the `schema_generation` marker. #062 lands here: the `schema_version` write becomes upsert (ON CONFLICT DO UPDATE) with a version-bump re-bootstrap test.
- **FR132-2 (backfill by replay):** The rebuild tool imports the live census (533 entities / 7 workspaces at PRD time — re-count at run) from the old file: every entity → an `entity_created` event + row; every `phase_events` row → an event on its axis; status/kanban derived ONCE at import via the current `derive_kanban` logic (prd.md:142) into the STORED v1-operational columns, then that logic is deleted (FR132-5a). Identity: fresh UUIDv7 minted per entity (PRD Migration Path; #054(b) pinned: re-mint ALL, no grandfathering); ALL uuid references (parent_uuid, entity_relations.from/to_uuid incl. 124's blocks edges, events.entity_uuid, workflow rows) remap through one old→new mapping applied atomically; the old uuid is preserved in the entity_created payload. Malformed legacy rows (empty ids, duplicate P001s) are deduped/normalized and LISTED in the report, never silently skipped (prd.md:106). Insert order pinned (127 GO caveat 1): each entity's event stream is written in ascending DOMAIN-timestamp order, so the views' `MAX(uuid)` latest-per-axis selection (uuid7 ≈ insert time on freshly-minted ids) resolves to the latest domain event, not the last-written row.
- **FR132-2b (axis mapping, pinned):** feature-kind phase events → `pipeline` axis (6-value PIPELINE_PHASES vocab, axes.py:65-67); 5D/brainstorm/backlog/task machine phases (the v17 CHECK admits discover/define/deliver/debrief etc., database.py:759,773) → `lifecycle` axis (vocab-free by design); status derivation → `execution` axis (derive_kanban outputs ⊆ EXECUTION_STATUSES — verified); `entity_created` → `lifecycle` axis. BEFORE import runs, every distinct census phase/status value is grepped against the live constraint surface (SC5's #077 clause).
- **FR132-3 (checksum/anomaly report):** Import emits a committed report artifact: counts per kind × workspace (old vs new, with equality or explained delta), dedupes performed, normalizations, orphans, uuid remap count, and a content checksum over a canonical serialization of both sides' comparable fields. Non-zero unexplained delta = tool exits non-zero.
- **FR132-4 (cutover):** After a successful backfill+report, the standard resolution path (`ENTITY_DB_PATH` default) points at the new file. Session-start reads use the per-entity `entity_axis_state` statement (127 GO §"statement-2": predicate pushes down; the pivoted `entity_state` full view is full-table-reads-only — #067 measured per-entity pivot reads O(total events)); the resolver's design constraint: candidate uuids come from a workspace-scoped entities query, THEN per-entity axis reads — the find-active join-then-filter shape (127 statement-1, a full events SCAN) is never on the hot path. Latest-event-per-axis semantics pinned BEFORE cutover: fresh uuid7 event ids are time-ordered; the sub-millisecond tie-break is defined and tested; backfill insert order per FR132-2. Success is reported only AFTER commit — closing the #055/#060 acknowledged-but-lost class, each pinned by a regression test shaped on its incident (backlog registration visible to get_entity + raw SQL immediately after success; phase_events row exists after complete_phase reports success).
- **FR132-4b (post-cutover events dual-write — OWNED, all five permitted writers classified):** the 109 doctor guard permits FIVE status/axis writers (check_status_write_path.py `_PERMITTED_ENCLOSING_DEFS`), not one; each is classified here and the design re-runs the #078 call-graph pass over all of them rather than asserting completeness:
  — `append_phase_event` (transitions, completes, 124's cascade flips): appends the corresponding `events` row IN THE SAME `BEGIN IMMEDIATE` as its v1 write.
  — `update_entity` (LIVE status mutations: abandon-feature workflow_state_server.py:2048, backlog archival cleanup_backlog.py:224, the update_entity MCP tool): its CURRENT shape commits the `UPDATE entities SET status` FIRST, then calls append_phase_event POST-COMMIT and FAIL-OPEN (database.py ~:7930-8020, swallowed to `audit_emit_failed_count`) — this violates both-or-neither and would silently reopen the #055/#060 class at cutover. FR132-4b directs: for status-changing calls the emit moves IN-transaction and FAIL-CLOSED (design chooses mechanics); an injected emit failure rolls back the status write.
  — `create_workflow_phase` (database.py:9178, initial-establishment INSERT, currently NO emit): owns the new entity's initial axis event, else post-cutover new entities diverge from replay (SC3).
  — `update_workflow_phase`/`upsert_workflow_phase`: presentational writers (kanban_column/mode) per the guard's own hint — EXCLUDED from event duty, exclusion recorded.
  Plus `register_entity`/`upsert_entity` (→ `entity_created`) and `delete_entity` (→ lifecycle event). #080 is absorbed here: wiring this path collapses the double-cascade-fire to ONE fire path — design chooses which survives (design-i1 correction: both fires are PYTHON call sites — update_entity's post-terminal cascade call and entity_engine's `_run_cascade`; no cascade SQL trigger exists, "trigger" was loose campaign vocabulary).

## #054 decisions pinned

- **(a) dropped-UNIQUE reliance:** the v2-generation entities table DROPS 109/129's composite `UNIQUE(workspace_uuid, type_id)` per P004 FR-4 — scoping is the `workspace_uuid` column + workspace-scoped queries + single-column indices (schema_v2.py:81-82), not a constraint. Consumers that leaned on v17 uniqueness are enumerated at design from the CALL GRAPH of `resolve_entity_uuid`/type_id-keyed lookups (per #078); each must tolerate duplicates via workspace scope or newest-wins with a warning. NOTE: at 132 the emitted entities table keeps the v1-operational shape incl. its composite `UNIQUE(workspace_uuid, type_id)` (Scope model step 1), so the drop bites only when the strip lands — the enumeration still happens NOW because backfill dedup decisions depend on it. Dedup trigger precision: the retained composite UNIQUE TOLERATES the census's known CROSS-workspace collisions (project:P001 ×3 workspaces, prd.md evidence) — forced dedup applies to WITHIN-workspace duplicates and malformed ids only; SC2's corpus models a within-workspace case.
- **(b) mixed uuid4/uuid7:** re-mint ALL as uuid7 at import; old uuid in payload for lineage.

## Hazards

- **H1 (FK remap atomicity):** re-minting uuids while 124's blocks edges + parent_uuid webs reference the old ones — the remap must be one transaction over the whole import; a partial remap is worse than no rebuild. SC2's idempotence run doubles as the torn-import probe.
- **H2 (constraint collisions at import — the #077 class):** the old DB contains 10 live status values incl. 13 empty-string rows (PRD evidence) and 5D/lifecycle phase vocabularies; FR132-2b's mapping + SC5's grep are the guard. Precision matters: 119's event_type CHECK is length-only, so the collision surface is to_value on the two constrained axes + the axis CHECK + relations kind — not event_type.
- **H3 (MCP startup caching):** servers cache DB path + workspace uuid at startup (prd.md:119; live-demonstrated by 124's stale-server incident) — cutover REQUIRES a session restart; the tool must print this loudly and the doctor marker must make a stale-path server detectable.
- **H4 (dark-suite reconciliation):** 120/126 test suites write arbitrary axis values that 122's now-active triggers reject — those fixtures move to vocabulary-legal values in the same change (roadmap:49); expect suite churn, bounded by the trigger vocabulary.
- **H5 (hooks/lib self-reference):** session-start hooks load database.py against whatever file exists — during the implementation window the repo's own session hooks run pre-cutover code; mirror 124's live-DB discipline (the rebuild tool runs only when invoked, never at module import).
- **H6 (#059 adjacency):** the fork-race flake lives in the OLD migration chain's tests; do not let new-file tests share its fixture pattern.
