# Design: 132-backfill-rebuild-tool

Implements spec.md (e70d2a7). Pointer-style; the spec's FR/SC/H text is binding — D-sections pin mechanics and resolve OQ-1/2/3.

## D1 — Build pipeline (Scope-model steps, connections pinned)

- **Step 1 (chain replay) = `EntityDatabase(staging_path)` construction, nothing more.** On an empty file `_migrate()` runs the full chain 1→19 (database.py:6210-6215, :10101-10112) — the sanctioned initial use the spec names; the tool then `.close()`s it. No hand-rolled replay loop.
- **Step 2 (selective v2 seed) = one raw `sqlite3.connect(staging_path)`** (corrected at design-i2). Registration mechanics: events.py:92 and views.py:95 register their DDL into `schema_v2.DDL_REGISTRY` at IMPORT time (module side effect); `register_vocab_ddl()` (axes.py:151) is the only register FUNCTION and must never run at import — step 2 therefore IMPORTS events/views (+ axes), CALLS `register_vocab_ddl()`, then executes the registry entries for owners `events`, `views`, `axes` (the entity_phase_status two-axis read view, axes.py:145 — SEEDED, it is the read surface later consumers adopt), `axes_vocab_triggers` (D7b's guard trigger ships INSIDE the `events` owner's DDL — no standalone register call exists) — NOT the `core` owner (step-1's chain already built those tables incl. `_metadata` at database.py:10095) and NOT `bootstrap_v2()` wholesale (IF-NOT-EXISTS would silently skip chain-shaped tables). One transaction. Requires SQLite ≥ 3.47 (register_vocab_ddl raises RuntimeError below — expression RAISE). sequences/workspaces/entity_relations keep chain shapes.
- **FK posture on the staging import connection (design-i2):** `PRAGMA foreign_keys=ON` set explicitly (a raw connection defaults OFF — the D2 topological order would otherwise be unenforced theater), AND `PRAGMA foreign_key_check` must return empty immediately before the import transaction commits (belt for the H1 one-transaction remap); non-empty → abort with the violating rows in the report.
- **Step 3 (stamp) = same raw connection**: upsert `schema_generation='v2'` + `schema_version=str(V2_SCHEMA_VERSION)` (#062's ON CONFLICT DO UPDATE lands in the shared upsert helper both generations use).
- **`_migrate()` generation guard (v1-layer edit, FR132-1):** placed immediately AFTER `_migrate`'s existing `CREATE TABLE IF NOT EXISTS _metadata` (database.py:10095 — the literal first statement; a guard placed before it would raise no-such-table on a pre-M1 file) and ABOVE the version loop: `SELECT value FROM _metadata WHERE key='schema_generation'`; `'v2'` → `return`. The SC1 phantom-migration-20 probe still discriminates guard vs empty-range.
- **Staging path:** `<dir>/entities.db.rebuild-<yyyymmdd>` beside the live file. The tool aborts if it already exists unless `--resume-check` re-verifies its report (idempotence per SC2). **Ordering dependency (design-i1): the migration-11 call-site edit (D6.5) ships WITH or BEFORE the tool** — until it lands, step-1's chain replay executes migration 11's `_atomic_write_workspace_mapping(workspace_root, {})` (database.py:1861) and pollutes CWD with the audit JSON (the #066 stray-writer, live-demonstrated in tests). Implement verifies no OTHER migration has an unguarded filesystem side effect on an empty file (grep os.\b/open( in migration bodies).

## D2 — Backfill algorithm (order + dedup + remap)

Read the OLD file via `sqlite3.connect("file:...?mode=ro", uri=True)` — never EntityDatabase (its construction mutates). Import order (FK-safe): workspaces → entities in parent-topological order (parent_uuid before children; cycle = anomaly row, imported parentless with note) → entity_relations → workflow_phases rows → phase_events (copied, uuid/type-remapped, `source='backfill'` — the CHECK admits it, database.py:5224-5239) → v2 `events` emission per D3. Per-entity uuid7 re-mint into one in-memory `old→new` map applied everywhere, PLUS a secondary `(workspace_uuid, type_id) → new_uuid` map for the business-key-keyed tables — phase_events and workflow_phases key on type_id/project_id and carry NO entity uuid (database.py:5219-5240), so their rows resolve through the secondary map; dedup LOSERS' phase_events/workflow_phases rows resolve to the SURVIVOR's new_uuid (mirroring the relations re-point) with an anomaly note (H1: whole import is ONE transaction on the staging connection).
Dedup (spec #054(a)): WITHIN-workspace `(workspace_uuid, type_id)` collision → keep newest `updated_at`, losers recorded as anomalies with their old uuids (their relations re-point at the survivor); cross-workspace collisions preserved. Empty/blank `entity_id`/`name` → normalized to `unnamed-<old-uuid-prefix>` + anomaly row. Every anomaly is a report entry, never a skip (prd.md:106).

## D3 — Event emission mapping (FR132-2b)

Every emitted event carries `actor='backfill:132'` (events.actor is NOT NULL length>0, events.py:72-83 — the v2 table has NO source column; actor IS the provenance). Bucketing is by kind AND event_type:

| Census source | axis | event_type | to_value |
|---|---|---|---|
| entity row (ALL kinds incl. bug/project/workspace) | lifecycle | entity_created | NULL (payload: old_uuid, kind, display fields) |
| phase-named phase_events of FEATURE-kind (event_type ∈ started/completed/skipped/backward AND phase ∈ 6-vocab) | pipeline | carried event_type | the phase (⊆ PIPELINE_PHASES) |
| phase-named phase_events of ALL OTHER kinds (project/task/brainstorm/backlog/bug — incl. 5D phases) | lifecycle | carried event_type | the phase (lifecycle axis is vocab-free, axes.py:122-131) |
| NON-phase phase_events rows (entity_status_changed, entity_promoted, entity_created-dupes, spawned_child, cascade_ready — phase typically NULL; historical status values may be OUT of the 122 vocab) | lifecycle | carried event_type | the carried value or NULL (NEVER the execution axis — legacy out-of-vocab values would trip the 122 trigger; history preserved without collision) |
| final derived status per entity (derive_kanban ONCE, prd.md:142) | execution | status_backfilled | the derived value (⊆ EXECUTION_STATUSES) |

SC3 parity note: the execution axis's latest event is BY CONSTRUCTION the synthesized status_backfilled row (inserted last per entity); pipeline's latest is the entity's last real phase event — NULL-to_value rows live on lifecycle only, so neither gating axis can have a NULL latest from backfill.

Pre-import (SC5 #077 clause): `SELECT DISTINCT` over every census phase/status/kind/relation-kind/derived-kanban/mode value, diffed against PIPELINE_PHASES/EXECUTION_STATUSES/the axis CHECK/the entities polymorphic CHECK/the phase_events vocab/`('fixes','blocks')`/the v1 workflow_phases kanban_column + mode CHECKs (database.py:753-777 — the derive-once values are WRITTEN there too) — mismatch aborts BEFORE any write, listing the offending values. Insert order within an entity: ascending domain timestamp (spec FR132-2; MAX(uuid) correctness).

## D4 — Cutover = rename swap (resolves OQ-1)

18 hard-coded `~/.claude/pd/entities/entities.db` literals + 8 `ENTITY_DB_PATH` readers exist; the swap touches NONE of them. Both files are WAL-mode (database.py `_set_pragmas`, schema_v2:182, events.py:108) — sidecars are first-class in the choreography: (0) staging connection `PRAGMA wal_checkpoint(TRUNCATE)` then `.close()` — its `-wal`/`-shm` are now empty/deleted, the main file carries the whole import; (1) `PRAGMA wal_checkpoint(TRUNCATE)` on the OLD file (fresh short-lived connection) then close, THEN `chmod a-w` + rename old → `entities.db.v1-readonly` (checkpointed ⇒ no live -wal; 30-day readers open `?mode=ro&immutable=1`, sidestepping WAL's -shm write-on-read); (2) rename staging → `entities.db` (verify no orphaned `entities.db.rebuild-*-wal` remains — abort if the checkpoint left one); (3) write marker `~/.claude/pd/migrations/v2-cutover.json` `{cutover_at, old_file, expiry: +30d, old_sha256, report_path}`; (4) print the H3 session-restart warning LOUDLY. Rollback = reverse renames (D10) — safe precisely because both sides were checkpointed. FR132-6 fail-loud: the read-only chmod makes any stale writer raise `sqlite3.OperationalError: attempt to write a readonly database` — the test asserts exactly that; the doctor gains a marker-aware check at 133 (not here — spec Non-goals).

### D4b — SC4 resolver readiness (owning section)

SC4 is a READINESS measurement on the backfilled v2 views, not a live read rewire (Scope model: post-cutover reads stay v1). FROM entity_axis_state WHERE entity_uuid = ?` for uuids obtained from a workspace-scoped `entities` query. `::test_resolver_explain_and_latency` builds the new file via the tool on a ~533-entity corpus, runs EXPLAIN QUERY PLAN on statement-2 (asserts index/pushdown, no `SCAN events` of the full table), and measures p95 over the corpus ≤ 5 ms per entity.

## D7b — FR132-7: events immutability vs INSERT OR REPLACE (design-i2 — was silently dropped)

**Choice: a BEFORE INSERT uuid-collision guard trigger** (`events_no_replace`: `BEFORE INSERT ON events WHEN EXISTS(SELECT 1 FROM events WHERE uuid = NEW.uuid) → RAISE(ABORT)`), registered with the step-2 DDL and added to events.py's trigger set for connect_v2 consumers. REJECTED alternative: `recursive_triggers=ON` — connection-scoped (every current and FUTURE connection must remember it; the existing events_no_delete BEFORE DELETE trigger only fires on REPLACE's implicit delete under that pragma, and neither `_set_pragmas` :10085-10090 nor connect_v2 sets it). The guard trigger is structural — fires on the INSERT half regardless of pragmas. Teeth test (D9): `INSERT OR REPLACE` on an existing event uuid raises; the row is byte-unchanged after.
