# Spec: State Projection Views (feature 120)

## Problem

The v2 track has an events log (119) but NO way to read current state from it: the v2 `entities` table deliberately carries no state columns ("NO status / workflow_phase / pipeline_phase / execution_status columns: state is events (119) projected via views (120)" — schema_v2.py:57-58), and nothing derives state from `events`. PRD FR-3 demands the projection be STRUCTURALLY unbypassable: "Current state is a projection of events, implemented as a VIEW over latest-event-per-axis. Raw SQL must not be able to bypass the event path" — closing feature-109's deferred runtime-enforcement hole (user story 2, prd.md:60-62) by construction (a VIEW is not writable; there is no state column to UPDATE), not by code discipline, triggers, or grep. The PRD success criterion requires a `replay(events) == projection` property test (prd.md:46). Two adjacent debts land here by prior agreement: backlog **#061** ("append_event factory-contract guard BEFORE feature 120" — a bare `sqlite3.connect` silently skips FK enforcement and can write a permanent orphan into the immutable events table; 121's dark rename helper already composes into append_event, so the guard closes a gap that predates 120 — and 120 is the first feature to READ/project state from events, where orphans would surface), and the roadmap's instruction to "capture the current session-start read-latency baseline for later comparison" (roadmap entry 7 — the number feature 126's OQ-1 keep-or-drop-`.meta.json` decision compares against).

## Scope

### In scope — DARK track (v2, guard-allowlisted, wired live at 132)
1. **State-projection VIEW(s):** new dark module `views.py` registering VIEW DDL through 118's `register_ddl` extension point (schema_v2.py:92), deriving current state as the latest event per `(entity_uuid, axis)`. "Latest" = MAX `uuid` per group BY DEFINITION — a deterministic rule, matching `read_events`'s canonical `ORDER BY uuid` (events.py:99/:104) by construction. Chronology caveat (do not let 122/125/126 assume otherwise): uuid7 string order is chronological for different-millisecond and same-process minting (lowercase-hex, timestamp most-significant-first), but CROSS-PROCESS same-millisecond appends order by independent random bits — deterministic, not wall-clock-meaningful. "Latest event" means MAX(uuid), not "most recent wall-clock write". View shape (pivoted vs per-axis rows vs both) is design D-1. The view must expose, per entity: each axis's current `to_value`, and enough event metadata for consumers to be useful without a second query (at minimum the winning event's `timestamp` per axis; exact column set is D-1). Value vocabularies are NOT constrained here — the view is value-agnostic (per-axis value CHECKs are feature 122's).
2. **Replay property test:** a stdlib-only, seeded pseudo-random generator (no new dependencies — the repo has no hypothesis and SC6-class neutrality forbids adding one) produces N randomized event sequences (interleaved entities, axes, values, actors); a Python replay function folds each sequence; the test asserts the SQL view equals the replay FIELD-BY-FIELD for every entity/axis, every case. Failure output must print the seed + the minimal failing sequence so a failure is reproducible (mechanism is design D-2).
3. **#061 factory-contract guard:** `append_event` gains a cheap guard rejecting connections that lack the `connect_v2` contract (`PRAGMA foreign_keys` probe — design D-3), converting the silent-orphan hazard into a loud error ON THE append_event PATH. Honesty about closure: this is runtime code discipline for the Python API only — a raw `INSERT INTO events` on a bare connection can still write a permanent orphan (SQLite cannot force per-connection FK enforcement structurally); that residual surface stays documented-not-defended. FR-3's STATE invariant (view + no state columns + event immutability) is the structural piece; the orphan guard is not and does not claim to be. The guard sits at `append_event` ENTRY, before the `conn.in_transaction` branch — both the standalone and caller-owns-transaction paths are covered by construction. Test plan: ADD a new test — `append_event(bare_conn, ...)` raises the guard error AND writes zero rows (`SELECT COUNT(*)` == 0, the before-any-write pin); plus the symmetric compose-path pin (bare conn with an already-open transaction → same raise, zero rows); PRESERVE `test_bare_connection_inserts_the_same_orphan_successfully` (test_events.py:506) unchanged — it is the raw-INSERT orphan pin documenting the residual surface, not a silent-skip pin to update.
4. **Ships-dark guard extension:** `views.py` joins `_V2_DARK_MODULES` with THREE needle spellings incl. the relative form (121's precedent — the 119 needle set is three-way) and seeded-offender teeth tests for ALL THREE spellings (121 shipped three separate teeth tests; the relative form is the likeliest false-negative).

### In scope — LIVE track (measurement only, zero behavior change)
5. **Session-start latency baseline capture:** run the existing worktree-comparison benchmark (`plugins/pd/hooks/tests/bench-session-start.sh`, feature-107 harness) and record the result in a committed artifact `docs/features/120-state-projection-views/latency-baseline.md`: median ms, sha, machine context, reproduction command, raw output. SCOPE HONESTY (what this number is): the harness runs session-start against an EMPTY `HOME=$(mktemp -d)` (bench script :55) — it measures fixed session-start OVERHEAD, not populated .meta.json/DB read latency, and yields a median only (not the p50/p95 NFR-3 names). This is deliberately the FOUNDATION snapshot (roadmap entry 7 says "capture the current baseline"); the realistic POPULATED-state p50/p95 read-latency baseline — the number PRD NFR-3 (prd.md:95) actually names — is explicitly ASSIGNED to feature 126 (it owns the OQ-1 keep-or-drop-.meta.json decision and is where a DB-direct read path first exists to measure), with 127 verifying against 126's number (roadmap.md:45); roadmap.md:95's "re-validated once DB-direct reads replace .meta.json reads" sanctions the split. Without this assignment the NFR-3 baseline would be owned by no feature. The artifact must state this scope so 126 doesn't mistake it for a read-latency number. No source files change.

## Success Criteria

- [ ] **SC1 — view correctness:** for a seeded entity with events on all three axes, the view returns each axis's latest `to_value` (uuid7-latest, not timestamp-string-latest — a deliberately out-of-order-timestamp fixture pins that uuid ordering wins); ALSO uuid-latest-not-INSERTION-latest: a direct-INSERT fixture writes a lexicographically LARGER uuid BEFORE a smaller one (insertion/rowid order ≠ uuid order), the two rows carrying DISTINCT `to_value`s (identical values would make the assertion vacuous whichever row wins), asserting the view returns the LARGER-uuid row's value — so a naive bare-column `GROUP BY` view (which SQLite resolves from an arbitrary/last-rowid row) FAILS while MAX(uuid) passes — without this, every in-process fixture is vacuous against the rowid confound (in-process uuid7 minting is insertion-ordered); an entity with events on ONE axis shows that axis only (pivoted shape: other columns NULL — exact assertion per D-1's shape); an entity with ZERO events is absent from the per-axis projection (and its pivoted row, if D-1 pivots from `entities`, shows all-NULL state); multiple events same axis → only the latest survives.
- [ ] **SC2 — structural non-writability:** `INSERT`/`UPDATE`/`DELETE` against the view raise `sqlite3.OperationalError` (pinned by test for all three verbs — SQLite views are read-only without INSTEAD OF triggers, and none are defined); the v2 `entities` table still has zero state columns (schema introspection pin: `PRAGMA table_info` column set unchanged from 118's DDL).
- [ ] **SC3 — replay property:** N ≥ 200 seeded random event sequences (multi-entity, multi-axis, shuffled interleavings) each satisfy `replay(events) == view` field-by-field; a failing case prints seed + sequence; the suite runs deterministically (fixed master seed) and within a non-regression time guard of < 5s wall for the property test on the dev machine (a guard, not an SLA; D-2 should reuse one bootstrapped DB/connection across cases — per-case bootstrap_v2 with its WAL+flock sidecar would dominate the budget).
- [ ] **SC4 — #061 guard:** NEW tests — `append_event` on a bare `sqlite3.connect` connection raises the guard's loud error BEFORE any write (`SELECT COUNT(*)` == 0 after), AND the symmetric compose-path case (bare conn with an already-open transaction) raises the same error, zero rows; `connect_v2` connections pass unchanged; `test_bare_connection_inserts_the_same_orphan_successfully` (test_events.py:506, the raw-INSERT residual-surface pin) is PRESERVED byte-unchanged; zero behavior change for every existing events test.
- [ ] **SC5 — baseline artifact:** `docs/features/120-state-projection-views/latency-baseline.md` exists with a measured median (ms), the sha measured at, machine context, a copy-pasteable reproduction command, the raw run output, AND an explicit scope statement (empty-state fixed overhead, median-only — NOT the NFR-3 p50/p95 read-latency baseline, whose CAPTURE is assigned to feature 126 and whose verification to 127); the number is derived from an actual run, not estimated.
- [ ] **SC6 — guard + neutrality:** ships-dark guard fails on live import of `views.py` (teeth test, all THREE spellings incl. relative); full `hooks/lib` + `mcp` + `ui` suites green; `validate.sh` 0 errors; doctor check count untouched; no new dependencies (stdlib-only property test); no MCP tool changes (tool count stays 20 — no docs-sync needed).

## Error & Boundary Cases

- Entity with zero events → absent from per-axis projection; pivoted-from-entities row (if D-1 chooses that shape) carries NULL state (a v2 entity pre-first-event is a legal transient).
- Two events, same axis, identical `timestamp` strings (cross-process same-millisecond appends) → MAX(uuid) is the tiebreak — deterministic, NOT wall-clock-ordered in that window (independent random bits); the out-of-order-timestamp fixture in SC1 covers the stronger case (later uuid, earlier timestamp).
- Event whose `to_value` is NULL (legal per 119's DDL — e.g. a lifecycle event clearing a value) → the view reports NULL as the current value (latest event wins even when its value is NULL; pinned by test — "latest non-null" would be a DIFFERENT, rejected semantic).
- Events for an axis never written by any entity → view yields no rows for that axis (no phantom axes).
- Orphan events (entity_uuid not in `entities`): the connect_v2 FK + the new #061 guard close the PYTHON-API bare-connection hole; raw SQL on a bare connection remains able to write an orphan (structurally uncloseable in SQLite — the preserved test_events.py:506 pin documents it). The view's join semantics for a present orphan (backfill bugs, raw writes) are design D-1's call — documented, not defended.
- `views.py` DDL registered but `bootstrap_v2` never called with the module imported → same factory-requirement contract as events.py's preamble (119 precedent): consumers must import DDL owners before bootstrap; 132 owns the canonical import-all sequence.
- Benchmark preconditions unmet (dirty tree — the script exits 2): the baseline capture happens at a clean commit; the artifact records the sha.

## Open Decisions (design resolves)

- **D-1 (view shape):** (b) per-axis view `entity_axis_state(entity_uuid, axis, to_value, event_uuid, timestamp)` — one row per (entity, axis); or (c) both, a pivoted `entity_state` defined over the per-axis primitive. A pivoted-ONLY shape is EXCLUDED — SC1's zero-events/one-axis/rowid-confound pins assert against per-axis rows, so the per-axis view must exist. Leaning: (c) — the per-axis view is the primitive (trivially correct, one GROUP BY/MAX), the pivoted view is the consumer-ergonomic face 122/125 will want; two small DDL statements, no duplication of the latest-wins rule.
- **D-2 (property-test mechanism):** stdlib `random.Random(master_seed)` driving per-case seeds; each case generates 1-8 entities × 0-12 events across random axes/values with out-of-order timestamps; replay = fold events per (entity, axis) keeping max-uuid. GENERATOR CONSTRAINT (rowid confound): a fraction of cases must insert events with SHUFFLED uuid order (pre-minted uuids inserted out of lexicographic order via direct INSERT) so insertion order diverges from uuid order — otherwise in-process minting makes uuid order == rowid order and a rowid-latest view passes every case. Print seed + sequence on failure. Leaning: exactly that — no shrinking machinery (sequences small enough to eyeball; a shrinker is YAGNI).
- **D-3 (#061 guard mechanism):** (a) `PRAGMA foreign_keys` probe at `append_event` entry (== 1 or raise) — checks the actual property that matters; (b) `connect_v2` sets a sentinel attribute the guard checks — cheaper per call but checks provenance, not the property. Leaning: (a), and it is effectively BINDING: option (b) would break the existing retry-path tests whose wrapper connections (`_FirstBeginImmediateLockedConnection`/`_AlwaysBeginImmediateLockedConnection`, test_events.py:371/:409) forward `.execute` but not arbitrary attributes — an attribute check AttributeErrors there, violating SC4's zero-behavior-change clause; a PRAGMA probe forwards through `.execute` and passes.
- **D-4 (baseline capture form):** run bench-session-start.sh as-is (it measures merge-base vs HEAD; on an unchanged-code branch both numbers estimate the same baseline) vs extracting its timing loop into a one-off. Leaning: run as-is, record BOTH numbers + note they estimate the same quantity. Design must also RESTATE the scope decision (empty-state/median = foundation snapshot here; populated p50/p95 CAPTURE = 126, verification = 127) so the artifact's scope statement has a design anchor, and add the one-line clarification to roadmap.md entry 8 (126-lossless-meta-json-projection) making the assignment durable outside this spec (edit already landed at roadmap.md:42).
