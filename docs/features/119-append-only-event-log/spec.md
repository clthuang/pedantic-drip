# Spec: Append-Only Event Log (feature 119)

## Problem

The v2 schema (118, dark-shipped) deliberately carries NO state columns on `entities` — state is to be events projected via views. The events table does not exist yet. Today's v1 `phase_events` (`database.py:1503-1520`) is phase-axis-only (`phase`, `event_type IN (started/completed/skipped/backward)`), keyed by business key (`type_id TEXT` + `project_id`), INTEGER-autoincrement PK, and is one of several write paths rather than the sole one. PRD FR-2 requires a generalized append-only `events` table as the ONLY write path for state; the roadmap scopes 119 to: create the table, enforce row immutability, and wrap writes in a shared BEGIN IMMEDIATE transaction.

Separately, 118's retro (action 2) assigned 119 ownership of the `bootstrap_v2` concurrency gap: two processes bootstrapping the same path concurrently failed with "database is locked" (`OperationalError`) ~50% (15/30 trials) — the docstring at `schema_v2.py:131-135` names 119 as the first concurrent consumer owning the fix.

## Scope

**Dark-shipped like 118:** nothing live reads or writes the new table until the 132 cutover. All new code lives in `plugins/pd/hooks/lib/entity_registry/` v2 modules. The v1 `phase_events` table, `append_phase_event` (`database.py:8352`), and every v1 write path are UNTOUCHED — generalization means the v2 design supersedes them at cutover, not that v1 is migrated now.

### In scope
1. **`events` table DDL** registered into 118's `DDL_REGISTRY` via `register_ddl("events", ...)` (`schema_v2.py:90`), applied by `bootstrap_v2` after core. Shape per PRD FR-2 + FR-4 (uuid7 PK; no business keys):
   - `uuid TEXT PRIMARY KEY` (uuid7 — provides both identity and time-order; no INTEGER autoincrement),
   - `entity_uuid TEXT NOT NULL REFERENCES entities(uuid)` (NO cascade per D-1's leaning — re-pin SC1's column set if design flips D-1),
   - `event_type TEXT NOT NULL CHECK(length(event_type) > 0)` (e.g. `entity_created`, `transition`, `display_renamed` — vocabulary open, NO value-set CHECK: 122/124 own vocabularies; the length CHECK only rejects empty strings per the P003 blank-fields precedent),
   - `axis TEXT NOT NULL CHECK(axis IN ('pipeline','execution','lifecycle'))` (the three-axis frame is PRD-level, fixed here; per-axis VALUE vocabularies are 122's),
   - `from_value TEXT` / `to_value TEXT` (nullable — creation events have no from),
   - `actor TEXT NOT NULL CHECK(length(actor) > 0)` (who/what appended: e.g. `mcp:workflow_state_server`, `backfill:132`),
   - `timestamp TEXT NOT NULL` (ISO-8601 UTC),
   - `payload TEXT` (JSON, nullable — carries FR-11 fidelity fields: iterations, reviewerNotes, skippedPhases, mode, branch...),
   - lookup indexes: `(entity_uuid, axis)` and `(timestamp)` (non-unique; FR-4 forbids UNIQUE on human-readable fields — none here are human-readable anyway).
2. **Row immutability, structural:** `BEFORE UPDATE` and `BEFORE DELETE` triggers on `events` that `RAISE(ABORT, ...)`. This does NOT conflict with PRD FR-3's "no triggers-as-guards" rejection — that rejection targets guarding a *writable state table* (state must instead be a view, 120's deliverable); the events table's own append-only property is AFFIRMATIVELY required by PRD NFR-4 ("Event rows are immutable (no UPDATE/DELETE path)") and FR-2 ("append-only"); triggers are SQLite's only structural mechanism for base-table immutability. What the PRD rejected (Approaches table) was a writable state table WITH trigger guards — a different animal. The triggers ship inside the same `register_ddl("events", ...)` script.
3. **Append API — the single writer:** new module `entity_registry/events.py` with `append_event(conn, *, entity_uuid, event_type, axis, from_value=None, to_value=None, actor, payload=None, timestamp=None) -> str` (returns the event uuid). Mints uuid7 via 118's `generate_uuid7()`; stamps UTC now when `timestamp` is None; `json.dumps` the payload dict. Runs inside the caller's transaction when one is open, else wraps itself in `BEGIN IMMEDIATE`/`COMMIT` with rollback on failure (the "shared BEGIN IMMEDIATE" of the roadmap: callers like 121's allocate-and-record or 120's projection-refresh compose multiple writes into ONE immediate transaction by opening it themselves; `append_event` must not force a nested/second transaction). NFR-1's `sqlite_retry` clause: the STANDALONE path wraps its BEGIN IMMEDIATE attempt with the shared `with_retry` (`plugins/pd/hooks/lib/sqlite_retry.py:27` — the F062 primitive NFR-1 names; already the `@with_retry("entity")` pattern in `server_helpers.py`); inside a caller-opened transaction, retry is the CALLER's responsibility (mirroring the composition rule — retrying mid-transaction would replay only a fragment). Explicitly NOT `EntityDatabase` methods — v2 modules stay import-light and side-effect-free like `schema_v2.py`/`uuid7.py` (118 pattern).
4. **Read API (minimal, for tests + 120):** `read_events(conn, entity_uuid, *, axis=None) -> list[dict]` ordered by `uuid` (uuid7 = time-ordered; sequential same-process appends are strictly ordered — a CPython stdlib `uuid.uuid7` property (module-global monotonic counter, single-threaded guarantee; verified by `test_uuid7.py:16-21`), NOT anything 118 built; cross-process ordering by uuid is best-effort and acceptable — single-user tool). One entity's full history = one query (PRD success criterion).
5. **v2 connection factory:** `connect_v2(db_path) -> sqlite3.Connection` in `events.py` (or a tiny shared `connection_v2.py` if design prefers) opening with `autocommit=True` (mirrors `bootstrap_v2`, `schema_v2.py:151` — makes `conn.in_transaction` the unambiguous open-transaction detector for `append_event`'s compose-or-wrap contract) and re-issuing `busy_timeout`, `journal_mode=WAL`, and `foreign_keys=ON` on every new connection — `schema_v2.py:137-142` explicitly documents that pragmas do NOT carry over from the bootstrap connection and assigns the factory to "119+". `append_event`/`read_events` document that they require a factory-configured connection (they take `conn`, they do not open paths).
6. **`bootstrap_v2` concurrency fix (118 retro action 2):** serialize concurrent bootstrap on the same `db_path` with an advisory file lock (`fcntl.flock` on `{db_path}.bootstrap.lock`, blocking with `busy_timeout`-comparable wait) held around the whole bootstrap body. Update the `schema_v2.py:131-135` docstring (the "119+ owns" sentence becomes the description of the lock). Reproduce 118's 30-trial concurrent harness as a regression test: 0/30 failures required.
7. **Ships-dark guard re-scope:** `TestSchemaV2ShipsDark.test_no_non_test_importer_of_schema_v2` (`test_schema_v2.py:500-533`) currently fails on ANY non-test hooks/lib file containing the string `schema_v2` — the feature's own `events.py` necessarily trips it (its docstring anticipated exactly this). Re-scope the guard to allowlist dark v2 SIBLING modules (`events.py`, future 120/122 registrants) while still failing on live-module wiring (`database.py`, engine, doctor, MCP servers). The guard's protective intent survives: v2 stays dark until 132; only the definition of "dark" widens from "no importers" to "no LIVE importers".
8. **Doctor/validate neutrality:** feature adds no live surface — doctor check count stays 19; `validate.sh` stays green; no README component counts change (docs-sync checklist: no doctor checks added/removed).

## Success Criteria

- [ ] **SC1 — table + registration:** `bootstrap_v2(path)` on a fresh path creates `events` with the exact FR-2 column set (verified via `PRAGMA table_info`), both lookup indexes, and both immutability triggers; `DDL_REGISTRY` contains owners `["core", "events"]` in order; double-import of `events.py` does not double-register (module-level registration is idempotent by module caching; a direct second `register_ddl("events", ...)` call raises ValueError per 118's contract — pin BOTH facts).
- [ ] **SC2 — append-only enforced structurally:** raw `UPDATE events SET to_value=...` and raw `DELETE FROM events` both raise `sqlite3.IntegrityError` with the trigger's message on a fresh BARE `sqlite3.connect` — no events.py Python in the write path (proves enforcement lives in the DB; the test MODULE imports events at top like its siblings, which is how the DDL registers — the purity claim is discharged at the CONNECTION level); `INSERT` succeeds. `DROP TABLE events` is not guarded (SQLite cannot; accepted — single-user tool, same acceptance as v1).
- [ ] **SC3 — single-writer API round-trip:** `append_event` writes a row readable by `read_events` with all fields intact (payload dict round-trips JSON); returned uuid is version-7/variant-correct (version assert mirrors `test_uuid7.py:20` `u[14]=='7'`; the variant assert `u[19] in '89ab'` is NEW — 118 never checked variant); omitted timestamp is stamped UTC-now ISO-8601; two SEQUENTIAL same-process appends are uuid-ordered = insertion-ordered (stdlib guarantee; no cross-thread ordering claim — SC4c needs only distinct uuids).
- [ ] **SC4 — transaction composition:** (a) `append_event` inside a caller-opened `BEGIN IMMEDIATE` does not commit/rollback the outer transaction (caller rollback discards the event); (b) standalone `append_event` on failure (e.g. FK violation on unknown `entity_uuid`) leaves no partial row and the connection usable; (c) two threads appending concurrently through their own connections both succeed under WAL + busy_timeout (small N, deterministic — not a stress test).
- [ ] **SC5 — bootstrap concurrency:** a NEW two-process concurrent `bootstrap_v2` harness (pattern from 118's exploratory trials, which were never committed) passes 30/30 (118 measured ~15/30 without the lock); the lock file is created next to the DB (per D-3's leaning — re-pin if design flips) and does not leak file descriptors (context-managed). (Uncontended single-process bootstrap takes the lock's fast path — rationale, not a measured criterion.)
- [ ] **SC6 — dark-ship neutrality:** full existing suites green (`hooks/lib`, `mcp`, `ui`); `validate.sh` 0 errors; doctor 19 checks with issue-class set unchanged (no capture bracket needed — feature touches no live check surface; assert count only); the re-scoped ships-dark guard passes WITH `events.py` present AND still fails on a seeded live-module reference (test the guard's teeth both ways); grep proves no live module imports `entity_registry.events` (dark-ship pin, non-vacuous once 132 flips it).

## Error & Boundary Cases

- `append_event` with unknown `entity_uuid` → FK violation surfaces as `sqlite3.IntegrityError` (workspaces/entities rows are prerequisites; FK enforcement is ON only on factory-configured connections (`schema_v2.py:139` — the pragma is per-connection and silently a no-op inside a transaction); pin a test that a `connect_v2` connection actually enforces the events FK, and a documenting test that a bare `sqlite3.connect` does NOT (the factory is load-bearing, not decorative)).
- `append_event` with invalid `axis` → CHECK violation raises; the open `event_type` deliberately accepts anything non-null.
- Payload that is not JSON-serializable → `TypeError` from `json.dumps` propagates BEFORE any SQL executes (no partial transaction state on the standalone path).
- Empty-string `actor`/`event_type` → rejected via CHECK (`length(actor) > 0`, `length(event_type) > 0`) — blank required fields helped corrupt the old DB (P003 precedent cited in PRD FR-5).
- Lock-file staleness: `flock` releases on process death (kernel-owned) — no stale-lock cleanup path needed; document this property in the docstring rather than building cleanup.
- Two `bootstrap_v2` calls in the SAME process/thread re-entering: the lock is per-fd and `flock` re-entry on a fresh fd blocks — bootstrap is module-init-time-once per process today; document, don't defend (single-user tool).

## Open Decisions (design phase resolves)

- **D-1 (FK delete semantics):** `entity_uuid` FK — plain `REFERENCES` (delete of an entity with events fails loud) vs `ON DELETE CASCADE` (audit trail dies with the entity) vs no FK (orphan events tolerated). Tension: PRD wants full audit reconstruction; v2 `entity_relations` chose CASCADE for dangling-rows-serve-nothing; events are NOT relations — an audit row about a deleted entity still serves auditing. Leaning: plain REFERENCES (fail loud; deletion flow at cutover appends a terminal `entity_deleted` event and soft-deletes, per PRD's auditability story) — but 132's backfill/delete semantics must confirm.
- **D-2 (payload schema discipline):** free JSON now vs a validated envelope. Leaning free JSON + a documented key registry in the module docstring (FR-11 fields), validation deferred to 120's projection which is the actual consumer.
- **D-3 (lock scope):** lock file beside db (`{db_path}.bootstrap.lock`) vs locking the db file itself. Leaning sidecar file — flocking the SQLite file itself interacts with SQLite's own POSIX locks.

## Verification Approach

Every SC maps to at least one test; SC2's DB-residency proof asserts on a raw bare `sqlite3.connect` (no events.py function in the write path; the test module imports events for DDL registration — connection-level purity, per design Testing #2). Keep runtime bounded (<~10s; small busy waits).
