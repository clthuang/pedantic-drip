# Spec: Two-Axis Phase/Status Schema (feature 122)

## Problem

PRD FR-6 (R2.5): today one conflated state field drives everything — `entities.status` is free TEXT (10 live values), `workflow_phases.workflow_phase`'s CHECK is the union of four vocabularies, and `derive_kanban(status, phase)` (kanban.py:26-40) translates the conflation into board columns with terminal-status-wins precedence. The v2 world needs the two axes SEPARATED with separate vocabularies and separate CHECKs: `pipeline_phase` (the authored-work pipeline: brainstorm→specify→design→create-plan→implement→finish; NULL for non-authored kinds) and `execution_status` (a small universal Kanban enum the board renders directly). Feature 120 shipped the per-axis state views with DELIBERATELY axis-generic column names (`pipeline_value`/`execution_value` — views.py's own comment: "feature 122 owns the two-axis value vocabularies and their live names"); 119's `events.to_value` is free TEXT with no per-axis vocabulary enforcement. 122 claims the names and ships the vocabularies + enforcement, dark.

**Roadmap correction required (entry 9):** the entry's "Delete the derive_kanban translation function" clause is NOT executable at 122 — derive_kanban has SIX live call sites across five files (workflow_state_server.py:959/:1234, backfill.py:335, reconciliation.py:201, feature_lifecycle.py, engine.py:620) and the live UI still renders v1 `kanban_column`. Deleting it before 123 (transition writes), 125 (UI reads), and 132 (cutover) would break the live board. 122 ships the REPLACEMENT semantics dark; the deletion moves to 132's cutover checklist (the 120/entry-8 precedent: roadmap edited at specify time, coordinated cross-feature).

## Functional Requirements

- **FR122-1 (pipeline vocabulary):** `PIPELINE_PHASES` = exactly `("brainstorm", "specify", "design", "create-plan", "implement", "finish")` — ordered, frozen, pinned by an exact-equality test. NULL allowed (non-authored kinds).
- **FR122-2 (execution vocabulary):** `EXECUTION_STATUSES` = the design-pinned universal enum (frozen, ordered, exact-pinned). It must be a SUPERSET of the six reachable `derive_kanban` outputs — each output a literal MEMBER, no translation anywhere (the named view is verbatim per FR122-4; any rename/drop of a board value would be a TRANSLATION and belongs to 132's backfill, not 122). Additions beyond the six are a design decision with recorded rationale (PRD OQ-3).
- **FR122-3 (DB-resident enforcement):** an out-of-vocabulary NON-NULL `to_value` on a `pipeline` or `execution` event is REJECTED at the DB layer with an error naming the axis AND the offending value (a design-phase "static-string RAISE" weakening was REVERTED 2026-07-12 — SQLite ≥3.47.0 accepts expressions in RAISE; empirically probed on this venv's 3.53.2 with the value interpolated via quote(NEW.to_value)); NULL passes; lifecycle is exempt (type_id-bearing renames pinned accepted). Enforcement fires on RAW INSERT too (structural — trigger class, not app-layer).
- **FR122-4 (the named face):** a registered view exposes per-entity `pipeline_phase`, `execution_status`, `pipeline_at`, `execution_at` under EXACTLY those names; values verbatim from the winning events (no translation); 120's shipped views/tests byte-untouched.
- **FR122-5 (ships dark):** new module allowlisted with needles + teeth; zero live importers; all existing suites green (the pre-122 green count, RE-DERIVED at implement time on the feature-base commit — not a pinned literal).
- **FR122-6 (roadmap coordination):** entry 9's deletion clause moved to 132's checklist in the same commit as this spec's approval; 125's entry gains "(derive_kanban deleted at 132, not before)" clarity.

## Error & Boundary Cases

- **Out-of-vocab with correct axis:** rejected, error names axis + value (SC2).
- **In-vocab value on the WRONG axis** (e.g., `to_value='wip'` on pipeline): rejected by pipeline's vocabulary check (wip ∉ pipeline) — pinned; the symmetric case (`'design'` on execution) likewise.
- **NULL to_value:** legal on all axes (120's NULL-latest semantics preserved); pinned.
- **Lifecycle carries a pipeline-vocab value** (e.g., legacy `completed` lifecycle events — 126's grammar uses them): ACCEPTED (lifecycle is vocab-free at 122); pinned so 123 inherits the decision consciously.
- **Existing dark-suite writers** (120's property test draws arbitrary to_values on all three axes; 126's generator likewise): 122's resolution IS the structural isolation — sibling suites never call register_vocab_ddl, so their DBs never carry the triggers and there is NO collision to resolve in the dark phase (pinned by the leak-detection test). Auditing those write-sites against then-present triggers is 132's named forward obligation at the canonical bootstrap, not 122 design work.
- **Trigger vs immutability trigger ordering:** the new BEFORE INSERT vocab trigger coexists with 119's BEFORE UPDATE/DELETE immutability triggers — no interaction (different verbs); pinned by running 119's immutability tests unchanged.
- **case sensitivity:** vocabularies are exact-case (`wip` not `WIP`); a wrong-case value is out-of-vocab — pinned.
