# Spec: Unified Transition Engine (feature 123)

**PRD:** P004 FR-7 (prd.md:85) — "A single per-kind transition-machine router replaces the split `ENTITY_MACHINES` + `WorkflowStateEngine` pair… Project-kind transitions and projections are ported instead of remaining the unowned special case (110/spec.md:36,48)." (Disambiguation: the workflow-rebuild PRD has its own unrelated FR-7 = express mode; every "FR-7" here means P004's.)

## Problem

Transition logic is split across three parallel implementations that disagree in shape and failure posture:
1. **WorkflowStateEngine** (engine.py:42) — feature kind; 4 ordered gates via transition_gate; fail-loud post-128 (WorkflowDBUnavailableError).
2. **FiveDBackend `_fived_transition`/`_fived_complete`** (entity_engine.py:468/:399) — initiative/objective/key_result/project/task; hand-rolled re-implementation of the gate logic; **two distinct silent-tolerate DB-error shapes** (degraded=True at :552-567; None+stderr at :452-457) — the KNOWN FR-10 violations 128 deferred here.
3. **ENTITY_MACHINES + transition_entity_phase** (entity_lifecycle.py:18/:124) — brainstorm/backlog lifecycle graphs; CRUD writes, no gates, no events.

Project kind is the unowned special case: its transitions ride the generic 5D path, its `.meta.json` is written once by `init_project_state` (feature_lifecycle.py:292-306) and never reprojected — and **any `transition_phase` on a `project:` id today would clobber that file with a feature-shaped payload**, permanently losing `features`/`milestones` (`_process_transition_phase` → `_project_meta_json` at workflow_state_server.py:1013-1014 has no kind gate; `_project_meta_json` :383-499 builds feature shape unconditionally). Latent (no current caller does this), but it is exactly the hazard FR-7's citation names.

## Functional Requirements

- **FR123-1 (one router):** A single per-kind transition registry/router owns transition validation and dispatch for ALL workflow kinds — feature (6-phase pipeline), the 5D set initiative/objective/key_result/task (5D phases), project (ported, owned), brainstorm and backlog (lifecycle graphs). The three parallel validation implementations collapse into it; `ENTITY_MACHINES` as a public standalone registry and `_fived_transition`'s hand-rolled gate logic are absorbed/deleted. Storage stays v1 (workflow_phases/entities CRUD via the sanctioned write helpers — the doctor `check_status_write_path` surface is unchanged); the router becomes the ONE choke point 132 later swaps storage under.
- **FR123-2 (project kind owned):** Project transitions are first-class in the router, and project **projections** are ported: a project-kind transition updates the project's `.meta.json` in PROJECT shape (id/slug/status/features[]/milestones[] preserved — 126 scoped its projector "feature-kind ONLY (project shape is 123's)"). The H1 clobber path dies: `transition_phase` on a `project:` id must never write a feature-shaped `.meta.json`. Red-first: pin today's clobber (features/milestones lost) before the fix.
- **FR123-3 (fail-loud everywhere — closes 128's deferral):** DB errors on ANY kind's mutation path raise `WorkflowDBUnavailableError` (or an equally structured, chained, transaction-rolling-back error) — BOTH 5D tolerate shapes convert: `_fived_transition`'s `degraded=True` return AND `_fived_complete`'s `None`+stderr return. The workflow_state_server.py:925 guard is deleted with its last producer; `TransitionResponse.degraded` is removed from models.py; the MCP `transition_phase` envelope drops its `degraded` key (a DELIBERATE schema change — the pinned 3-key test at test_workflow_state_server.py:1329-1345/:1378 updates to the 2-key shape; README/tool-doc sweep for any `degraded` mention).
- **FR123-4 (legacy producer retired — closes 125's handoff):** The brainstorm `reviewing` phase maps to `wip`, not `agent_review` (the last live producer of a dropped v1 column value). `LEGACY_VALUE_REMAP`'s `agent_review` entry becomes defensive-only (same posture as `human_review` — stored rows only; comment updated, entry still DELETED at 132). The two literal pins update: test_workflow_state_server.py:6584-6589 and the UI producer-union test (test_deepened_app.py:1001-1027 — which must STAY green by construction since wip ∈ EXECUTION_STATUSES).
- **FR123-5 (invariants preserved):** bug/task remain status-only (NO machine entries — the tested negative space at test_status_only_lifecycle.py:49-50 and the anti-widening guard at test_entity_lifecycle.py:416-426 survive or are consciously revised in the same change). MCP tool surfaces (`transition_phase`, `complete_phase`, `transition_entity_phase`, `init_entity_workflow`) keep their names and input contracts; the ONLY envelope change is FR123-3's `degraded` removal. The `kind` vs `entity_type` read inconsistency inside entity_engine.py (:153/:229 vs :281/:404/:475) collapses onto `kind` (the DB aliases both; result dicts continue to carry both keys outward).
- **FR123-6 (express-mode representation — resolves PRD OQ at prd.md:196):** Per-kind phase validity IS the representation: each kind's valid phases/transitions live in its router machine definition — no second vocabulary, no per-(kind,mode) axis. Express-mode phase subsets are represented by the EXISTING kind-agnostic `skipped` event seam (workflow_state_server.py:972-995, already un-gated on kind) + `metadata.skipped_phases`. The workflow-rebuild track consumes this; its `mini_spec` event is NOT 123's (zero grep hits today; that track ships after this one).

## Open Questions (design phase)

- **OQ-1 (router placement + import direction):** workflow_engine/ vs entity_registry/ module home; entity_lifecycle.py's MCP-imported functions (`_lib_init_entity_workflow`/`_lib_transition_entity_phase` at workflow_state_server.py:43-44) — absorb-and-re-export vs move; no import cycles (axes.py:44 already assigns any future lifecycle vocabulary here).
- **OQ-2 (project projection mechanism):** kind-gate `_project_meta_json`'s call site + a parallel project-shaped projector vs one kind-dispatching projector; interaction with 127's deny-by-default guard (the sanctioned-writer allowlist is pinned by an exact-4 set-equality test — a new writer symbol REQUIRES an allowlist edit + test update, or the projector must route through the existing `_project_meta_json` symbol renamed/generalized).
- **OQ-3 (machine definition shape):** one declarative schema expressive enough for 6-phase-pipeline + 5D + lifecycle graphs (transitions/forward/columns/gates vary), vs per-kind strategy classes behind one router protocol.
- **OQ-4 (gate reuse):** the 4 transition_gate gates (G-18/G-08/G-23/G-22) are feature-tuned — which apply per kind, and does the 5D hand-rolled skip/backward logic map onto them or stay per-kind gate sets?
