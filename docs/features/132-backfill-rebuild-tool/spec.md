## Scope model (load-bearing)

The new DB file is born from ONE canonical all-DDL bootstrap emitting the UNION of both generations, with the v1-operational shape winning wherever both define a table:
- **v1 operational DDL at its migration-19 shape** — entities WITH status (database.py:173-184), workflow_phases WITH kanban_column (database.py:402-415), phase_events, entities_fts, entity_relations (124's `('fixes','blocks')` CHECK), all live indices/triggers. v1 access-layer code keeps reading/writing these against the new file.
- **v2 event core seeded from schema_v2.py** — events + entity_axis_state/entity_state views (120), sequences, _metadata, workspaces. bootstrap_v2 (schema_v2.py:130) seeds ONLY this half; its stripped entities shape (schema_v2.py:43-58, deliberately status-less) is NOT the emitted entities table at 132 — P004 FR-4's column-strip is explicitly deferred to later v1-retirement work.
- **122's vocabulary triggers ACTIVE from birth** (axes.py events_vocab_pipeline/execution) — reconciling 120/126's dark-suite arbitrary-value writes in the same change (roadmap:49).

Post-cutover, state mutations remain on the v1 write path (which the triggers police) AND land as events; per-consumer v2-read rewires beyond this spec's inventory are NOT in scope (133 owns doctor; later work owns v1-path retirement). The old file is retained read-only for 30 days as the escape hatch.

## Functional Requirements

- **FR132-1 (rebuild tool + version bookkeeping):** A committed, idempotently re-runnable rebuild tool stands up the NEW DB file (path distinct from `~/.claude/pd/entities/entities.db`) via the canonical all-DDL bootstrap above. Version bookkeeping: `_metadata` carries `schema_generation='v2'` + `schema_version=V2_SCHEMA_VERSION` (schema_v2.py:16) as the ONE version location. Because `EntityDatabase.__init__` unconditionally runs `_migrate()` (database.py:6210-6215) which replays `range(current+1, max(MIGRATIONS)+1)` (database.py:10101-10112) — i.e. the whole v1 chain at version 1 — `_migrate()` gains an IN-SCOPE generation guard: a `schema_generation='v2'` file returns immediately; the v1 chain is v1-generation-only. (Setting `schema_version=19` instead was rejected: future v1 migrations 20+ would silently auto-run against the v2 file.) New-file fingerprint = the `schema_generation` marker. #062 lands here: the `schema_version` write becomes upsert (ON CONFLICT DO UPDATE) with a version-bump re-bootstrap test.
- **FR132-2 (backfill by replay):** The rebuild tool imports the live census (533 entities / 7 workspaces at PRD time — re-count at run) from the old file: every entity → an `entity_created` event + row; every `phase_events` row → an event on its axis; status/kanban derived ONCE at import via the current `derive_kanban` logic (prd.md:142) into the STORED v1-operational columns, then that logic is deleted (FR132-5a). Identity: fresh UUIDv7 minted per entity (PRD Migration Path; #054(b) pinned: re-mint ALL, no grandfathering); ALL uuid references (parent_uuid, entity_relations.from/to_uuid incl. 124's blocks edges, events.entity_uuid, workflow rows) remap through one old→new mapping applied atomically; the old uuid is preserved in the entity_created payload. Malformed legacy rows (empty ids, duplicate P001s) are deduped/normalized and LISTED in the report, never silently skipped (prd.md:106).
- **FR132-2b (axis mapping, pinned):** feature-kind phase events → `pipeline` axis (6-value PIPELINE_PHASES vocab, axes.py:65-67); 5D/brainstorm/backlog/task machine phases (the v17 CHECK admits discover/define/deliver/debrief etc., database.py:759,773) → `lifecycle` axis (vocab-free by design); status derivation → `execution` axis (derive_kanban outputs ⊆ EXECUTION_STATUSES — verified); `entity_created` → `lifecycle` axis. BEFORE import runs, every distinct census phase/status value is grepped against the live constraint surface (SC5's #077 clause).
- **FR132-3 (checksum/anomaly report):** Import emits a committed report artifact: counts per kind × workspace (old vs new, with equality or explained delta), dedupes performed, normalizations, orphans, uuid remap count, and a content checksum over a canonical serialization of both sides' comparable fields. Non-zero unexplained delta = tool exits non-zero.
- **FR132-4 (cutover):** After a successful backfill+report, the standard resolution path (`ENTITY_DB_PATH` default) points at the new file. Session-start reads use the per-entity `entity_axis_state` statement (127 GO §"statement-2": predicate pushes down; the pivoted `entity_state` full view is full-table-reads-only — #067 measured per-entity pivot reads O(total events)). Latest-event-per-axis semantics pinned BEFORE cutover: fresh uuid7 event ids are time-ordered; the sub-millisecond tie-break is defined and tested. Post-cutover mutation writes append events + projection in ONE `BEGIN IMMEDIATE` (NFR-1); success is reported only AFTER commit — closing the #055/#060 acknowledged-but-lost class, each pinned by a regression test shaped on its incident (backlog registration visible to get_entity + raw SQL immediately after success; phase_events row exists after complete_phase reports success).
- **FR132-5 (deletion/rewire inventory — all in this feature, post-backfill):**
  (a) `derive_kanban` (workflow_engine/kanban.py:25) + its SIX call sites: workflow_state_server.py:1003/:1272, backfill.py:335, reconciliation.py:201, feature_lifecycle.py:205, engine.py:529 — each rewired to read the STORED axis values (entities.status / workflow_phases.kanban_column via 125's `execution_status` alias), consistent with the Scope model; comment/docstring stragglers swept (axes.py:70, constants.py:4, router.py:70, board.py:28 + import lines).
  (b) Migration-11 shims — both PRD anchors have DRIFTED; live symbols: the PRD's "database.py:5902 compatibility shim" is `_resolve_workspace_uuid_kwargs` (database.py:6285, docstring :6296 "Compatibility shim for Feature 108 Migration 11 transition window") — delete it + the legacy `project_id` kwarg surface on its callers and the entity_server tool params (grep-enumerate at design); plus `_atomic_write_workspace_mapping` (database.py:1726 — also closes #066's root cause).
  (c) `LEGACY_VALUE_REMAP` (ui/routes/helpers.py:144) — deletable because backfill normalizes stored values to the 122 vocab (125 handoff).
  (d) #054(c): `create-project.md` P{NNN} filesystem scan → atomic allocator; backfill seeds v2 `sequences` for EVERY kind from the census max; remove `allocate_entity_id`'s `entity_type="project"` rejection guard in the same change.
  (e) #075: rewire `transition_entity_phase` enforcement onto `get_machine(kind).validate()` for lifecycle kinds and delete the redundant moved-legacy path (one enforcement route per kind, matching 123's 5D rewire).
- **FR132-6 (old-DB window):** The old file is made read-only (chmod + a doctor-visible marker recording the cutover date and the 30-day expiry); the tool never deletes it. Any code path that would WRITE the old path post-cutover fails loud.
- **FR132-7 (#068):** The new file's events immutability holds against `INSERT OR REPLACE` (enable `recursive_triggers` on connect to the new file, or a BEFORE INSERT uuid-collision guard trigger — design chooses), with a teeth test proving REPLACE cannot silently delete+reinsert an event.

## Success Criteria

- **SC1:** Fresh-bootstrap test: tool creates the new file from empty; `_metadata` carries `schema_generation='v2'` + the single `schema_version` cell (#062 upsert test included); `EntityDatabase(new_file)` construction runs ZERO v1 migrations (generation guard test — assert the chain's marker writes are absent); 122 vocab triggers fire on the new file from birth (arbitrary-value INSERT rejected).
- **SC2:** Backfill parity: on a seeded corpus modeling the live census's pathologies (empty ids, duplicate P001s, uuid4 rows, blocks edges, multi-workspace, 5D + lifecycle phase histories), the report shows per-kind×workspace equality (or enumerated deltas), every anomaly listed, checksum stable across two runs (idempotence: second run no-ops or reproduces byte-identical report).
- **SC3:** Replay property (prd.md:107): for every imported entity, replaying its event stream reproduces the projection field-by-field; divergence fails the test naming the entity and field.
- **SC4:** Cutover reads, non-vacuous by pinned shape: the session-start resolver issues the per-entity `entity_axis_state` statement (127 GO statement-2); EXPLAIN QUERY PLAN on THAT statement shows the entity_uuid predicate pushdown (no full-view SCAN of events); p95 ≤ 5 ms per entity at the re-censused ~533-entity scale (127 measured μs — 5 ms is the flake-proof ceiling). 127's full-scan walk analogue may be re-run as INFORMATIONAL only; it does not gate (its 32 ms @ N=533 vs the N=22-derived 31 ms threshold is the conflation this SC previously invited).
- **SC5:** Deletion grep-gate: zero live references to `derive_kanban`, `_resolve_workspace_uuid_kwargs`, `_atomic_write_workspace_mapping`, `LEGACY_VALUE_REMAP`, and the legacy `project_id` kwarg outside git history/this feature's artifacts (exemption list pinned at design). #077 clause: every value the backfill writes into a constrained column OR trigger-policed axis (`to_value` on pipeline/execution) is grepped against the LIVE constraint definition first — the constraint surface is exactly: `event_type` CHECK length>0 (events.py:76 — NO vocabulary), `axis IN ('pipeline','execution','lifecycle')` (events.py:77), 122's to_value triggers on pipeline+execution only (axes.py:122-131; lifecycle vocab-free), `entity_relations.kind IN ('fixes','blocks')`.
- **SC6:** Old-DB safety: post-cutover, a write attempt against the old path fails loud (test); the read-only marker carries the dated expiry.
- **SC7:** #055/#060 regression pins green (FR132-4's two tests) — the write path cannot report success before commit.

## Hazards

- **H1 (FK remap atomicity):** re-minting uuids while 124's blocks edges + parent_uuid webs reference the old ones — the remap must be one transaction over the whole import; a partial remap is worse than no rebuild. SC2's idempotence run doubles as the torn-import probe.
- **H2 (constraint collisions at import — the #077 class):** the old DB contains 10 live status values incl. 13 empty-string rows (PRD evidence) and 5D/lifecycle phase vocabularies; FR132-2b's mapping + SC5's grep are the guard. Precision matters: 119's event_type CHECK is length-only, so the collision surface is to_value on the two constrained axes + the axis CHECK + relations kind — not event_type.
- **H3 (MCP startup caching):** servers cache DB path + workspace uuid at startup (prd.md:119; live-demonstrated by 124's stale-server incident) — cutover REQUIRES a session restart; the tool must print this loudly and the doctor marker must make a stale-path server detectable.
- **H4 (dark-suite reconciliation):** 120/126 test suites write arbitrary axis values that 122's now-active triggers reject — those fixtures move to vocabulary-legal values in the same change (roadmap:49); expect suite churn, bounded by the trigger vocabulary.
- **H5 (hooks/lib self-reference):** session-start hooks load database.py against whatever file exists — during the implementation window the repo's own session hooks run pre-cutover code; mirror 124's live-DB discipline (the rebuild tool runs only when invoked, never at module import).
- **H6 (#059 adjacency):** the fork-race flake lives in the OLD migration chain's tests; do not let new-file tests share its fixture pattern.
