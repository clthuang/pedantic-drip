# Specification: Workspace-Scoped Queries

## Problem Statement

The live entity DB's query surface leaks across workspaces by default — the Kanban board renders every workspace's cards mixed together (`ui/routes/board.py:59` calls `list_workflow_phases()` with zero args; the method has no workspace parameter at all, `database.py:9038`), `query_ready_tasks` is unscoped end-to-end with no override, and workspace filtering for the two `list_features_by_*` MCP tools exists only as a post-filter bolt-on in the server layer — while cross-workspace links are policed by a heavyweight allowlist mechanism (`cross_workspace_allowlist` table + `CrossWorkspaceError` gate + doctor check + a "grandfather" fix action that was never even registered in the fixer) that P004 FR-9 explicitly retires in favor of ordinary uuid references.

## Acceptance Criteria

### Happy Paths

- Given entities in workspaces A and B, when `add_dependency`/`set_parent`/`add_okr_alignment` link an A-entity to a B-entity, then the operation succeeds and the link reads back — no `CrossWorkspaceError`, no allowlist involvement.
- Given features in workspaces A and B, when the Kanban board route renders with the current workspace resolved to A, then only A's cards appear.
- Given `list_features_by_phase` called with no `project_id`, when the server's workspace is A, then only A's features return for all NON-orphan rows (contract preserved, mechanism pushed down) — with ONE declared change: orphan `workflow_phases` rows (entity missing), which the old post-filter dropped as a side effect, are now RETURNED (intended: anomaly visibility, consistent with the board's orphan-retention AC); with `project_id='*'`, all workspaces return.
- Given ready tasks in workspaces A and B, when `query_ready_tasks` runs under workspace A, then only A's tasks return; with `'*'`, both.
- Given a project root matching exactly one `workspaces` row, when `check_unknown_workspace_orphans` or `check_entity_orphans` runs, then behavior is identical to pre-extraction (helper is pure refactor).

### Error & Boundary Cases

- Given a project root matching zero or multiple `workspaces` rows, when the adopted helper returns None, then both checks fall back exactly as today (unfiltered / no-resolvable-workspace branches preserved — 0 and >1 were already collapsed in both).
- Given the DB is unhealthy and `list_by_phase`/`list_by_status` fall back to the filesystem scan (`engine.py:222-249`), then results are implicitly current-workspace-only (local artifacts_root) — `'*'` cannot be served in degraded mode; accepted for single-user degraded reads, noted so it is not later mistaken for a scope bug.
- Given the doctor runs post-deletion, then zero `cross_workspace_parent_uuid` issues are emitted, the check-count regression tests pin 19, and no other check's issue class drifts (same-session before/after capture).
- Given a dangling `parent_uuid` crossing workspaces created before this feature (previously "unallowlisted"), when doctor runs, then it is simply NOT flagged (ordinary reference per FR-9) — coverage for genuinely broken refs (parent uuid missing entirely) remains with the existing orphan checks.
- Given `list_workflow_phases(workspace_uuid=None)` called directly (no scope), then it returns all rows (unscoped default preserved at the DB layer — the DEFAULTING to current workspace lives at MCP/UI/engine callers, matching today's `_resolve_optional_workspace_filter` convention).
- Given a `workflow_phases` row whose entity is MISSING (orphan — the LEFT JOIN at `database.py:9070-9077` yields NULL workspace), when `list_workflow_phases(workspace_uuid=W)` filters, then the orphan row is RETAINED (predicate `(e.workspace_uuid = ? OR e.uuid IS NULL)`) — orphans belong to no workspace and must stay visible on the board (today they render under 'backlog'; a naive equality filter would silently hide a known, fixer-handled anomaly class — the disappearance cousin of vacuous-green).

## Feasibility Assessment

### Assessment
**Overall:** Confirmed
**Reasoning:** Every mechanism already exists in-tree: `_resolve_optional_workspace_filter` is the established optional-scope pattern (8 call sites), the MCP default-to-current-workspace resolution exists (`workflow_state_server.py:1974-2016`) and merely moves from post-filter to parameter, and the deletion sweeps mirror 131's `check_project_attribution` deletion (same deregistration shape). The gate removal is pure deletion — the PRD settled the policy question.
**Key Assumptions:**
- The cross-workspace enforcement surface is COMPLETELY enumerated — Status: Verified, with the method stated honestly: the behavior grep (`workspace_uuid !=` + `cross.workspace.*forbidden`) surfaced issue_spawn (`entity_server.py:751` single-line message) but NOT the complete_phase closes gate — its message spans lines and its comparison is `row.get("workspace_uuid") !=`, defeating both patterns. That gate was found by reviewer code-reading of the closes-handling path; the full-set claim rests on reviewer sweep of every `closes`/link-creating write path plus the reversal-test census (`CrossWorkspaceError` appears in exactly one test file; the two inline gates have exactly one rejection test each). Lesson recorded: text grep — even behavior-shaped — cannot bound an enforcement surface; path-reading of every link-creating write is the closing step.
- No consumer branches on `error_type="cross_workspace_forbidden"` — Status: Verified (grep over py/md/js/html: zero hits outside the emitting sites).
- The board/entities UI has no other unscoped LIST/SEARCH reads beyond the 5 cited call sites — Status: Verified at design phase (full sweep of `plugins/pd/ui/` for `db.`/`_db.` query-method calls). The sweep found exactly FOUR additional reads, all in the entity DETAIL route (`entities.py:146` `get_entity`, `:160`/`:163` `get_lineage` up/down, `:167` `get_workflow_phase`) — single-entity reads on an already-identified entity, which SC6's boundary explicitly leaves unscoped by design (scope applies to LIST/SEARCH surfaces; detail navigation arrives FROM the now-scoped list; `get_entity`'s pre-existing cross-workspace type_id ambiguity error is unchanged behavior).
- `_lookup_workspace_uuid_by_project_root` (str|None) is NOT directly reusable for the checks (they need the 0/1/>1 list shape) — a doctor-local list-returning helper is the right extraction — Status: Verified (helper conflates 0 and >1; checks branch on the distinction).
**Open Risks:** The reversal surface spans three test sites (gate matrix + issue_spawn + complete_phase closes); enumerated up front per the 118 lesson. Interim UX window: between 129 and 130 the board shows ONLY the current workspace with no switcher — recorded decision, PRD-aligned (per-workspace default is the desired end state; single-user tooling).
