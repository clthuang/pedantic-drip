# Spec: Unified Transition Engine (feature 123)

**PRD:** P004 FR-7 (prd.md:85) — "A single per-kind transition-machine router replaces the split `ENTITY_MACHINES` + `WorkflowStateEngine` pair… Project-kind transitions and projections are ported instead of remaining the unowned special case (110/spec.md:36,48)." (Disambiguation: the workflow-rebuild PRD has its own unrelated FR-7 = express mode; every "FR-7" here means P004's.)

## Problem

Transition logic is split across three parallel implementations that disagree in shape and failure posture:
1. **WorkflowStateEngine** (engine.py:42) — feature kind; 4 ordered gates via transition_gate; fail-loud post-128 (WorkflowDBUnavailableError).
2. **FiveDBackend `_fived_transition`/`_fived_complete`** (entity_engine.py:468/:399) — initiative/objective/key_result/project/task; hand-rolled re-implementation of the gate logic; **two distinct silent-tolerate DB-error shapes** (degraded=True at :552-567; None+stderr at :452-457) — the KNOWN FR-10 violations 128 deferred here.
3. **ENTITY_MACHINES + transition_entity_phase** (entity_lifecycle.py:18/:124) — brainstorm/backlog lifecycle graphs; CRUD writes, no gates, no events.

Project kind is the unowned special case: its transitions ride the generic 5D path, its `.meta.json` is written once by `init_project_state` (feature_lifecycle.py:292-306) and never reprojected — and **any `transition_phase` on a `project:` id today would clobber that file with a feature-shaped payload**, permanently losing `features`/`milestones` (`_process_transition_phase` → `_project_meta_json` at workflow_state_server.py:1013-1014 has no kind gate; `_project_meta_json` :383-499 builds feature shape unconditionally). Latent (no current caller does this), but it is exactly the hazard FR-7's citation names.

## Functional Requirements

- **FR123-1 (one router):** A single per-kind transition registry/router owns transition validation and dispatch for ALL machine-bearing kinds. The per-kind machine table (code-verified):

  | Kind | Machine class | Source today |
  |---|---|---|
  | feature | 6-phase pipeline, WEIGHT-AGNOSTIC (the enforced machine is transition_gate.PHASE_SEQUENCE — engine.py:14/:28; templates.py's feature rows are runtime-dead data, tests-only; weight variance = the skipped-events overlay, FR123-6) | WorkflowStateEngine |
  | initiative, objective, key_result | 5D templates, per-(kind, weight) | FiveDBackend + templates.py |
  | task | 5D templates (standard: define→deliver→debrief; light: deliver — templates.py:40-41) | FiveDBackend (entity_engine.py:46 routes it) |
  | project | 5D templates — PORTED to owned status (FR123-2) | FiveDBackend (the unowned case) |
  | brainstorm, backlog | lifecycle graphs | ENTITY_MACHINES (entity_lifecycle.py:18) |
  | **bug** | **NONE — status-only, the sole machine-less kind** (entity_engine.py:46 excludes it; database.py:67 bug_flow) | — |

  Eight machine-bearing kinds + bug (= the nine VALID_ENTITY_TYPES, database.py:5780-5786; the helper map _KIND_TO_TYPE_LIFECYCLE's tenth entry, workspace→'none', is not a registerable kind and never transitions — excluded by construction). The three parallel validation implementations collapse into the router; `ENTITY_MACHINES` as a public standalone registry and `_fived_transition`'s hand-rolled gate logic (entity_engine.py:508-546) are absorbed/deleted. Storage stays v1 (workflow_phases/entities CRUD via the sanctioned write helpers — the doctor `check_status_write_path` surface is unchanged); the router becomes the ONE choke point 132 later swaps storage under.
- **FR123-2 (project kind owned):** Project transitions are first-class in the router, and project **projections** are ported: a project-kind transition updates the project's `.meta.json` in PROJECT shape (id/slug/status/features[]/milestones[] preserved — 126 scoped its projector "feature-kind ONLY (project shape is 123's)"). - **New concurrency machinery** — the router inherits the v1 write helpers' `BEGIN IMMEDIATE`/sqlite_retry semantics (PRD NFR-1) unchanged.
