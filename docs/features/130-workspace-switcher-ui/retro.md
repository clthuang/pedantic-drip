### Tune (Process Recommendations)

1. **Verify dispatch-briefing figures against primary sources before enshrining them in a downstream artifact.** Signal: three independent, source-checkable discrepancies in one "orchestrator-supplied, verified" briefing (observation 1). Recommendation: extend CLAUDE.md's reviewer-claim-verification and author-restated-literal guardrails one layer up — any agent receiving a prepared context bundle for synthesis should re-derive headline counts from `.review-history.md`/`.meta.json`/git before restating them. Confidence: **high**.
2. **Add a Testing-Strategy symmetric-branch checklist line to design/plan-reviewer.** Signal: a boundary condition fully described in prose with two complementary branches had a test for only one, surviving 4 dispatches across 3 reviewer types (observation 3). Recommendation: whenever a boundary condition is stated with symmetric/complementary branches, require the Testing Strategy to name a distinct test per branch — a single test titled after the shared condition is the tell that only one branch is covered. Confidence: **medium** (one confirmed occurrence at 130, but it's a specific instance of the "prose claim, unverified until an execution-shaped check runs" class 119's retro already generalized for API/schema claims).
3. **Track code-quality-reviewer's UI-track fix-rate before changing its gating.** Signal: 2 of 3 battery members produced a fix this feature; code-quality-reviewer produced zero (observation 2). Recommendation: don't drop or downgrade it yet — its nits are legitimate signal even unfixed (e.g., the duplicated `_option_element` test helper) — but track its fix-rate across the next 2-3 UI-track (routes/templates-only) features; if it stays at zero actionable findings, consider a lighter/summary-only pass for that track while keeping it fully gating on DB/engine-track work. Confidence: **medium** (n=1 feature for this precise per-reviewer breakdown).
4. **Keep normalizing campaign-trend comparisons by scope-class, not raw sequence.** Signal: 130 (3 blockers, ~2h24m42s) and 119 (2 blockers, ~2h24m) — the campaign's two most narrowly-scoped features — sit far below 129 (9 blockers, ~5h23m), while 131/118 (7/5) fall between. Recommendation: no process change — this validates, not contradicts, 119's retro's own "scope confound" finding; keep reading blocker-count/cycle-time trends by scope-class rather than as a monotonic learning curve. Confidence: **high** (directly measured against both 130's own data and the two prior retros' figures).
5. **Give the orchestrator a concrete mechanism for recovered-but-unrecorded implement-phase incidents.** Signal: a second confirmed occurrence of 129's named anti-pattern — dispatch-report findings invisible to `.review-history.md` (observation 5). Recommendation: add a lightweight "Process Notes" subsection to `.review-history.md` at implement-phase completion for recovered, non-blocking mid-phase incidents (stray processes, self-clobbered edits), rather than relying on the orchestrator to relay them out-of-band per-retro. Confidence: **high** (recurrence confirms it's a gap, not a one-off).

### Act (Reflections)

**Patterns:**
- Scoping a feature to explicitly exclude adjacent surfaces (spec.md: "no MCP changes, no engine changes, no write paths") lets every design decision be checked against installed library source rather than assumed — design-reviewer's iteration-1 pass verified `TestClient` cookie mechanics, `RedirectResponse`+`set_cookie`, and `uuid.UUID`'s 32-hex leniency directly against the installed package, closing on 0 blockers because there was nothing left to assume. (Feature 130, design phase; confidence: high)
- Spelling a control-flow restructuring byte-exact across design, plan, and tasks (the try/except closing before the HX/full-page branch split, D5) converged plan-reviewer and task-reviewer on the identical two gaps independently, and the restructured code then shipped with zero implement-phase blockers on that exact path — the same "front-load skeptic rounds" effect 129's retro validated for deletions, now shown to scale to control-flow surgery. (Feature 130, create-plan → implement; confidence: high)

**Anti-patterns:**
- A boundary condition's two complementary behaviors (route-side rejection vs. read-side fallback, both for "malformed uuid") were fully described in spec.md and design D5's prose, yet the Testing Strategy pinned a test for only the write-side branch — invisible through spec-reviewer, plan-reviewer, and two task-reviewer iterations, caught only when relevance-verifier diffed spec SC3 against the literal enumerated test list. (Feature 130, create-plan phase, relevance-verifier round 1; confidence: high)
- An orchestrator-supplied "verified" context bundle handed to a downstream analysis agent can itself carry unverified, source-contradicting figures (an iteration count, a campaign trajectory with two features' totals swapped, a reviewer-breakdown miscount) — the same drift class CLAUDE.md's guardrails already police for reviewer↔artifact and artifact↔artifact literals, now observed one layer up at the dispatch-briefing↔artifact boundary. (Feature 130 retro compilation; confidence: high)

**Heuristics:**
- When a design/plan states a runtime behavior as two symmetric branches off one boundary condition, treat the Testing Strategy as incomplete until you can point to a distinct test name for each branch — a single test titled after the shared condition is the tell that only one branch is actually covered. (confidence: high)
- Before citing a headline metric (iteration count, blocker count, campaign trajectory) supplied by an upstream dispatch or briefing, re-derive it from the primary source (`.review-history.md`, `.meta.json`, git) rather than repeating the briefing's prose — cost nothing to check here and caught three separate discrepancies. (confidence: high)
